# TUI Load Older Messages on Scroll

## Problem

The TUI only renders the last ~100 messages because:
1. Initial sync fetches with `limit: 100` (`sync.tsx:704`)
2. Live event eviction removes the oldest message when count exceeds 100 (`sync.tsx:392`)
3. No pagination logic exists to fetch older messages

The server API already supports cursor-based pagination via `limit` + `before` query params, returning `X-Next-Cursor` header when more data exists.

## Design

### Approach: Polling-based scroll detection + cursor pagination

Since OpenTUI's `<scrollbox>` does not expose a scroll event callback, we'll use a `setInterval` inside `onMount` that periodically checks `scroll.y` (the scroll position). When the user scrolls near the top (within ~2 screen heights), we trigger a load of the next 30 older messages using the `before` cursor.

### Key Decisions

1. **Load 30 messages per page** — reasonable batch size for terminal rendering
2. **Track loading state** — `loadingOlder` signal prevents duplicate concurrent loads
3. **Cursor handling** — Read `X-Next-Cursor` from the raw SDK response headers (`response.headers.get("X-Next-Cursor")`). The SDK client returns `{ data, response }` where `response` is the raw fetch Response.
4. **Prepend to store** — Older messages are prepended to `store.message[sessionID]` and parts to `store.part[messageID]`
5. **Eviction threshold increased** — From 100 to 500 to give headroom for auto-loaded pages
6. **Scroll position preservation** — After prepending, measure the scroll height delta and adjust via `scroll.scrollBy()` to keep the user's view stable
7. **Sticky scroll stays at bottom** — The `stickyStart="bottom"` behavior is unchanged; auto-load only triggers when user manually scrolls near the top

d) **Add `loadOlderMessages` method** to the `session` object:
```ts
async loadOlderMessages(sessionID: string) {
  const cursor = nextCursors.get(sessionID)
  if (!cursor) return { hasMore: false }

  const result = await sdk.client.session.messages({ sessionID, limit: 30, before: cursor })
  const items = result.data ?? []
  if (items.length === 0) {
    nextCursors.delete(sessionID)
    return { hasMore: false }
  }

  const newCursor = result.response?.headers?.get?.("X-Next-Cursor")
  if (newCursor) nextCursors.set(sessionID, newCursor)
  else nextCursors.delete(sessionID)

  // Prepend to store
  setStore(produce((draft) => {
    const existing = draft.message[sessionID] ?? []
    const newMessages = items.map(x => strip(x.info))
    draft.message[sessionID] = [...newMessages, ...existing]
    for (const msg of items) {
      const existingParts = draft.part[msg.info.id]
      if (!existingParts) draft.part[msg.info.id] = msg.parts
    }
  }))

  return { hasMore: !!newCursor, count: items.length }
}
```

c) **Add scroll polling in `onMount`** — Add a `setInterval` that checks scroll position:
```ts
let scrollInterval: ReturnType<typeof setInterval> | undefined

onMount(() => {
  scrollInterval = setInterval(() => {
    if (!scroll || scroll.isDestroyed) return
    if (!hasMoreOlder() || loadingOlder()) return

    // Check if user is near top (within 2 screen heights)
    const nearTop = scroll.y < scroll.height * 2
    if (!nearTop) return

    setLoadingOlder(true)
    const beforeHeight = scroll.scrollHeight

    sync.session.loadOlderMessages(route.sessionID).then((result) => {
      setHasMoreOlder(result.hasMore)
      // Preserve scroll position
      if (scroll && !scroll.isDestroyed) {
        const afterHeight = scroll.scrollHeight
        scroll.scrollBy(afterHeight - beforeHeight)
      }
    }).finally(() => {
      setLoadingOlder(false)
    })
  }, 200)
})
```

### 3. Edge Cases

| Scenario | Handling |
|---|---|
| User at bottom, new messages arrive | Existing `stickyScroll` handles this; auto-load only triggers near top |
| Rapid scrolling to top | `loadingOlder()` signal prevents duplicate requests |
| No more messages | `hasMoreOlder` becomes false, auto-load stops |
| Session switch | `createEffect(on(() => route.sessionID))` resets `hasMoreOlder` and `loadingOlder` |
| Empty session | No content to scroll, `hasMoreOlder` stays true but no trigger |
| Eviction during live updates | Threshold at 500 gives ~16 pages of headroom |

### 4. Cursor Fallback

If the SDK response does not expose headers properly, construct the cursor from the oldest message:

The server-side cursor format is `Buffer.from(JSON.stringify({ id, time })).toString("base64url")` where `time` is `time_created` (a finite number >= 0, epoch milliseconds). We can replicate this client-side:

```ts
function encodeCursor(id: string, time: number): string {
  // In the TUI runtime (Bun/Deno), we can use Buffer or btoa
  return Buffer.from(JSON.stringify({ id, time })).toString("base64url")
}
```

Use the oldest message's `id` and `time.created` to construct the `before` cursor. This is a fallback if `X-Next-Cursor` header is not accessible.

## Risks & Mitigations

- **SDK header access**: If `response.headers` is not available from the v2 SDK, use cursor encoding fallback
- **Performance**: Eviction at 500 provides a safety net. OpenTUI's `<scrollbox>` may have `viewportCulling` option if needed later
- **Scroll property**: Use `scroll.y` (consistent with existing session/index.tsx usage at line 395) not `scroll.scrollTop`
- **Sticky scroll interference**: `stickyStart="bottom"` only activates when user is at the bottom; manual scrolling near top disables it, so prepending won't fight with auto-scroll
