# Plan: merge-script opencode reference worktrees

## Goal

Extend `script/upstream/merge.ts` instead of adding a separate script. The merge script already knows the target opencode version, transformed opencode branch, base branch, and post-automation conflict state, so it is the right place to create prompt-ready reference worktrees.

## Worktree layout

Use the repo-local ignored `.worktrees/` directory, scoped to this workflow:

```text
.worktrees/opencode-merge/
  opencode/
  kilo-main/
  auto-merge/
```

- `opencode/`: pristine upstream opencode at the requested `--version` or `--commit`; the version is not in the directory name.
- `kilo-main/`: Kilo base branch snapshot, normally `main`/`origin/main`; if `--base-branch` is used, this should point at that base while the fixed path stays the same.
- `auto-merge/`: committed snapshot of the merge result after the script's automated conflict handling. If conflicts remain, conflict markers are committed as ordinary file contents on a local snapshot branch so an agent can inspect the exact automated state without touching the real merge branch.

## Key feasibility note

Yes, the third worktree is feasible, but it should not be created by committing unresolved conflicts on the real merge branch.

Git normally cannot commit while the real index has unmerged entries. Even if `git add -A` would turn conflict markers into normal content, doing that in the real merge worktree would mutate the user's actual merge state. Instead, create the snapshot commit using a temporary index:

1. Leave the real merge index untouched.
2. Load `HEAD` into a temporary index with `GIT_INDEX_FILE=<temp> git read-tree HEAD`.
3. Stage the current working tree into that temporary index with `GIT_INDEX_FILE=<temp> git add -A`.
4. Write a tree with `GIT_INDEX_FILE=<temp> git write-tree`.
5. Create a local commit with `git commit-tree <tree> -p <kilo-head> -p <opencode-branch> -m ...`.
6. Check `.worktrees/opencode-merge/auto-merge` out at that snapshot commit/branch.

That gives the agent a normal committed git tree containing the automated result, including any remaining conflict markers, while the real merge branch remains unresolved and ready for manual resolution.

## Implementation changes

1. Add helper code to `script/upstream/merge.ts` or a small `script/upstream/utils/worktree.ts` module.
   - Prefer a helper module if the worktree logic is more than a few small functions.
   - Keep the existing merge flow intact and only call the helper at the conflict/success endpoints.
2. Add `MergeOptions.worktrees`, defaulting to enabled.
   - Add `--no-worktrees` as an escape hatch for CI or users who only want branches/reports.
   - Do not run this logic during `--dry-run` or `--report-only`.
3. Add/update usage text in `merge.ts` and `script/upstream/README.md`.
   - Document the fixed paths and explain that `auto-merge` may intentionally contain conflict markers.

## Worktree helper behavior

Shared helper responsibilities:

- Ensure `.worktrees/opencode-merge` exists.
- Run `git worktree prune` before checking existing paths.
- If a target path exists, confirm it is a git worktree.
- Before reusing an existing target worktree, require `git -C <path> status --porcelain` to be clean.
- Never delete arbitrary directories automatically.
- Use detached worktrees for `opencode` and `kilo-main` to avoid branch collisions.
- Use a local snapshot branch for `auto-merge`, for example `opencode-merge/auto-v1.1.49`, so the committed snapshot is retained.

Specific refs:

- `opencode`: `targetVersion.commit` from upstream before Kilo transformations.
- `kilo-main`: the pulled base branch commit captured immediately before creating the Kilo merge branch.
- `auto-merge`: a commit-tree snapshot built from the current working tree after automated conflict handling.

## Merge flow integration

Capture these values during the existing merge script:

1. After `git checkout(config.baseBranch)` and `git pull(config.originRemote)`, capture `baseSha = git rev-parse HEAD`.
2. After `opencodeBranch` is created and transformed, capture `opencodeSha` or use the branch as the second parent for the auto snapshot.
3. After Step 7 automated conflict handling finishes:
   - If manual conflicts remain, create all three worktrees before saving final conflict instructions and before `process.exit(1)`.
   - If all conflicts auto-resolve, create all three worktrees before the final summary.
4. If later finalization regenerates lock files or SDK files, decide whether `auto-merge` should represent the state before or after finalization. Recommended: before finalization, because it is the output of automated merge/conflict resolution, not follow-up generated artifact commits.

## Output

At the end of the merge script, print a dedicated section:

```text
Reference worktrees:
  opencode:   .worktrees/opencode-merge/opencode      (<tag>, <sha>)
  kilo-main:  .worktrees/opencode-merge/kilo-main     (<base>, <sha>)
  auto-merge: .worktrees/opencode-merge/auto-merge    (<snapshot branch>, <sha>)

Agent prompt:
  Use these references while resolving the merge:
  - upstream opencode: <absolute opencode path>
  - Kilo base main: <absolute kilo-main path>
  - automated merge snapshot: <absolute auto-merge path>
```

## Safety details

- Do not push the `opencode-merge/auto-*` snapshot branch by default.
- Do not commit conflict markers on the real Kilo merge branch.
- If `auto-merge` already has user edits, fail with a clear message instead of overwriting it.
- If the merge has non-textual conflicts, the temporary-index snapshot still records the current filesystem state; the merge report and conflict list remain the source of truth for which conflicts need attention.
- Keep the real merge branch's unresolved index untouched when conflicts remain.
