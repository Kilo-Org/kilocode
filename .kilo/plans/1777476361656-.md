# Plan: opencode comparison worktree script

## Goal

Add a Bun script under `scripts/api` that prepares fixed git worktrees for:

- upstream opencode at a caller-specified version/tag/commit
- Kilo `main` at its current remote/main state

The output should be stable paths that can be pasted into an agent prompt during upstream merge conflict resolution.

## Directory decision

Use the repo-local `.worktrees/` directory, which is already ignored by the root `.gitignore`. This is easy and preferable because:

- worktrees remain near the repo and are easy to reference with stable relative paths
- `.worktrees` is already excluded from git, so creating large checkout directories will not dirty status
- git supports nested sibling worktrees inside an ignored directory without copying full repository history

Recommended layout:

```text
.worktrees/upstream-refs/
  opencode/
  kilo-main/
```

Do not include the opencode version in the worktree directory name. The script can print the resolved version/commit so the fixed `opencode` path remains stable while still making the checked-out ref explicit.

## Files to add/change

1. Add `scripts/api/opencode-worktrees.ts`.
   - `scripts/api` does not currently exist, so create it.
   - Keep the script standalone instead of modifying shared upstream merge code.
2. No changes required to `.gitignore` because `.worktrees` is already ignored.
3. Optional, if the repo expects script discoverability in `package.json`: add a root package script such as `"api:opencode-worktrees": "bun run scripts/api/opencode-worktrees.ts"`. This is not necessary if direct execution is acceptable.

## Script behavior

Command examples:

```bash
bun run scripts/api/opencode-worktrees.ts --version v1.1.49
bun run scripts/api/opencode-worktrees.ts v1.1.49
bun run scripts/api/opencode-worktrees.ts --commit abc123
```

Implementation outline:

1. Parse args:
   - accept positional version/tag as the common case
   - accept `--version <version>` where `1.1.49` and `v1.1.49` both work
   - accept `--commit <sha>` for non-tag refs
   - accept optional `--remote <name>` defaulting to `upstream`
   - accept optional `--main <ref>` defaulting to `origin/main`
   - accept optional `--base-dir <path>` defaulting to `.worktrees/upstream-refs`
2. Validate environment:
   - run from inside a git worktree
   - require the upstream remote to exist; if missing, print `git remote add upstream git@github.com:anomalyco/opencode.git`
   - require the main ref to resolve; use `origin/main` rather than local `main` so it works even when local `main` is checked out elsewhere or absent
3. Fetch refs:
   - `git fetch <remote> --tags --force`
   - `git fetch origin main`
4. Resolve upstream ref:
   - for `--commit`, verify `git rev-parse --verify <sha>^{commit}`
   - for version/tag, normalize by trying `<input>`, `v<input>` when missing `v`, and `<remote>/<input>` only if needed
   - resolve to a commit hash with `git rev-parse --verify <ref>^{commit}`
5. Prepare directories:
   - ensure `.worktrees/upstream-refs` exists
   - if `.worktrees/upstream-refs/opencode` or `.worktrees/upstream-refs/kilo-main` already exists, update it rather than fail
   - call `git worktree prune` before checking for stale records
6. Create/update worktrees:
   - for opencode: detached worktree at `.worktrees/upstream-refs/opencode` for the resolved upstream commit
   - for main: detached worktree at `.worktrees/upstream-refs/kilo-main` for `origin/main` (or `--main`)
   - prefer detached worktrees to avoid branch name collisions and to keep the checkouts read-only reference snapshots
   - if a target worktree already exists, use `git -C <path> fetch ...` then `git -C <path> checkout --detach <commit>` after ensuring it has no uncommitted changes
   - if a target path exists but is not a git worktree, fail with a clear message instead of deleting it
7. Output fixed prompt-friendly paths:
   - absolute path to opencode worktree
   - absolute path to Kilo main worktree
   - resolved upstream ref/tag/commit
   - resolved main commit
   - a suggested prompt snippet for the merge agent

## Safety details

- Never remove existing directories automatically.
- Before updating an existing worktree, check `git -C <path> status --porcelain`; fail if dirty.
- Use Bun `$` tagged template commands with argument interpolation to avoid shell quoting issues.
- Use short single-word locals where practical (`root`, `dir`, `ref`, `sha`, `path`) to match repo style.
- Keep comments minimal; only explain detached worktree behavior if needed.

## Validation

Plan for implementation verification:

1. `bun run scripts/api/opencode-worktrees.ts --help` or invalid args should print usage without stack traces.
2. If upstream remote is absent, the script should fail with the exact setup command.
3. With upstream configured, run a dry functional check using a real tag, for example:
   `bun run scripts/api/opencode-worktrees.ts --version v1.1.49`
4. Verify:
   - `git worktree list --porcelain` includes `.worktrees/upstream-refs/opencode`
   - `git worktree list --porcelain` includes `.worktrees/upstream-refs/kilo-main`
   - main repo `git status --short` does not show generated worktree contents because `.worktrees` is ignored

- No changeset is needed unless this script is considered user-facing release functionality; it is repo automation.
