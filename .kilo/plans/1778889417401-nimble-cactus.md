# TUI Message Auto-Load on Scroll

## Problem

The TUI currently loads only the first 100 messages on initial sync (`sync.tsx:704`) and never loads older messages. When the user scrolls to the top, they see nothing beyond what was initially loaded. The server API already supports cursor-based pagination via `limit` and `before` query params, and `MessageV2.page()` returns a `more` flag and `cursor` for loading additional pages.

## Current State

### Data Layer
- **sync.tsx** (`session.sync()`): Fetches messages with `limit: 100` — `sdk.client.session.messages({ sessionID, limit: 100 })`
- **Server API** (`session.ts:642-704`): Supports `limit` + `before` (cursor) query params. Returns messages array with `X-Next-Cursor` / `Link` headers when more data exists.
- **MessageV2.page()** (`message-v2.ts:1108-1143`): Returns `{ items, more, cursor }` — cursor-based pagination.
- **In-memory eviction** (`sync.tsx:391-409`): When `message.updated` event arrives and store has >100 messages, the oldest is evicted.

### UI Layer
- **session/index.tsx**: Renders messages via `<For each={messages()}>` inside a `<scrollbox>`.
- **Scroll**: OpenTUI `<scrollbox>` with `stickyStart="bottom"`. Access via `scroll` ref (`ScrollBoxRenderable`).
- **Scroll properties**: `scroll.y` (current scroll position), `scroll.scrollHeight` (total content height), `scroll.height` (viewport height), `scroll.getChildren()` (rendered children with `y` positions).
- **No scroll event listener**: There is currently no mechanism that detects when the user scrolls near the top.

### SDK
- **v2 SDK** (`sdk/js/src/v2/gen/types.gen.ts`): `SessionMessagesData` supports `limit?: number` and `before?: string` query params.
- The response is `Array<{ info: Message, parts: Part[] }>`. The cursor is returned in response headers (`X-Next-Cursor`), not in the body.

## Design

### Approach: Polling-based scroll detection via `createEffect`

Since OpenTUI's `<scrollbox>` does not expose a scroll event callback, we'll use a SolidJS `createEffect` that periodically checks `scroll.y` (the scroll position). When the user is near the top (within ~2 screen heights), we trigger a load of the next 30 older messages.

### Key Decisions

1. **Load 30 messages at a time** — matches the user's request.
2. **Track loading state** — prevent duplicate concurrent loads with a `loadingOlder` signal.
3. **Track oldest message ID** — use the first (oldest) message's ID to construct the `before` cursor. Actually, we need the cursor, not just the ID. The server returns the cursor in `X-Next-Cursor` header. We need to store this.
4. **Cursor handling** — The SDK returns only the body (messages array). The cursor is in the response header. We need to either:
   - Access the raw response to get the header, OR
   - Construct the cursor ourselves from the oldest message's `id` and `time_created`.

   Looking at `MessageV2.cursor.encode({ id, time })` — the cursor is a base64-encoded `{ id, time }` struct. We can reconstruct it from the oldest loaded message.

   **Better approach**: The server returns `X-Next-Cursor` header. The generated SDK client may expose headers. Let's check if the SDK response includes headers or if we need to use a lower-level call.

   Actually, looking at the sync code, it uses `sdk.client.session.messages({ sessionID, limit: 100 })`. The SDK is generated by openapi-ts and returns `{ data, response }` where `response` is the raw fetch response. We can read `response.headers.get("X-Next-Cursor")`.

   **Simpler approach**: Since we control the server and know the cursor format, we can encode the cursor from the oldest message's metadata. The `MessageV2.cursor` is exported from `message-v2.ts` but that's server-side only. Instead, we'll store the `nextCursor` from the server response header when available.

5. **Prepend to store** — New older messages are prepended to `store.message[sessionID]` and their parts to `store.part[messageID]`.
6. **Remove eviction or adjust threshold** — The current eviction at 100 messages will need adjustment. Since we're now loading on demand, we should either:
   - Increase the eviction threshold (e.g., 500), OR
   - Only evict when total exceeds a higher limit, OR
   - Remove eviction entirely and let the user's scroll naturally manage what's visible.

   **Decision**: Increase the eviction threshold to 500. This gives plenty of headroom for auto-loaded pages (500 / 30 ≈ 16 pages) while still preventing unbounded memory growth.

7. **Preserve scroll position** — When prepending messages, the scroll position should stay visually anchored to the message the user was looking at. Since we're adding content above, we need to measure the height difference and adjust `scroll.scrollBy()` accordingly.

## Implementation Plan

### 1. Modify `sync.tsx` — Add `loadOlderMessages` function and increase eviction threshold

**File**: `packages/opencode/src/cli/cmd/tui/context/sync.tsx`

- Increase eviction threshold from `100` to `500` (line 392).
- Add a `loadOlderMessages(sessionID: string, beforeCursor?: string)` method to the `session` object that:
  - Calls `sdk.client.session.messages({ sessionID, limit: 30, before: beforeCursor })`
  - Reads the `X-Next-Cursor` header from the response
  - Prepends messages to `store.message[sessionID]` and parts to `store.part[...]`
  - Stores the next cursor for subsequent loads
  - Returns `{ hasMore: boolean, nextCursor: string | undefined }`

### 2. Modify `session/index.tsx` — Add scroll monitoring and auto-load

**File**: `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx`

- Add signals:
  - `const [olderCursor, setOlderCursor] = createSignal<string | undefined>()` — tracks the cursor for loading older messages
  - `const [loadingOlder, setLoadingOlder] = createSignal(false)` — prevents concurrent loads
  - `const [hasMoreOlder, setHasMoreOlder] = createSignal(true)` — tracks if more messages exist

- After initial sync completes (in the `createEffect` at line 271), capture the `X-Next-Cursor` header if available and set it via `setOlderCursor()`.

- Add a `createEffect` that:
  - Uses `setInterval` (inside `onMount`) to poll `scroll.scrollTop` every 200ms
  - When `scroll.scrollTop < scroll.height * 2` (user is within 2 screen heights of top) AND `hasMoreOlder()` AND NOT `loadingOlder()`:
    - Calls `sync.session.loadOlderMessages(route.sessionID, olderCursor())`
    - Updates `olderCursor`, `hasMoreOlder`, and `loadingOlder` signals
    - After messages are prepended, adjusts scroll position by measuring the height difference

- The scroll polling should be debounced/throttled to avoid excessive checks. A 200ms interval is reasonable.

### 3. Handle scroll position preservation

When older messages are prepended, the content above the current viewport grows. To keep the user's view stable:

- Before loading: capture `scroll.scrollHeight` as `beforeHeight`
- After loading completes: capture new `scroll.scrollHeight` as `afterHeight`
- Adjust: `scroll.scrollBy(afterHeight - beforeHeight)` to shift the viewport down by the amount of new content
- Note: Use `scroll.scrollTop` (not `scroll.y`) for reading the current scroll position

### 4. Update the eviction logic in sync.tsx

Change the threshold from `100` to `500` at line 392:
```ts
if (updated.length > 500) {
```

This ensures we don't aggressively evict messages that the user just scrolled to load.

## Edge Cases

1. **User at bottom when new messages arrive** — The existing `stickyScroll` behavior handles this. Auto-load only triggers when near the top.
2. **Rapid scrolling to top** — The `loadingOlder` signal prevents duplicate requests.
3. **No more messages** — `hasMoreOlder` becomes false, auto-load stops triggering.
4. **Session switch** — The existing `createEffect(on(() => route.sessionID, ...))` resets state.
5. **Empty session** — No messages to load, `hasMoreOlder` stays true but no content triggers scroll-to-top.
6. **Cursor encoding** — We rely on the server's `X-Next-Cursor` header, which is already implemented.

## Risks

- **OpenTUI scroll access**: `scroll.scrollTop` is the correct property for reading scroll position (not `scroll.y`, which is the element's layout Y position). We'll poll via `setInterval` since there's no scroll event callback.
- **Performance**: Rendering many messages in a terminal UI could be slow. The eviction at 500 provides a safety net. If needed, we could add viewport culling later, but OpenTUI's `<scrollbox>` already has a `viewportCulling` option.
- **SDK response headers**: Need to verify the generated SDK exposes response headers. If not, we can construct the cursor from the oldest message's `id` and `time.created` using the same encoding the server uses.
