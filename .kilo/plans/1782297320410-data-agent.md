# Add the built-in Data agent

## Goal

Add a visible native primary agent named `data` for notebook-first data work. The initial version should stay deliberately small: a dedicated prompt, Code-like permissions, and no bundled rules, skills, MCP configuration, model, variant, temperature, or step limit.

## Decisions

- Use the agent key `data`; do not add a `data-analyst` alias.
- Set `mode: "primary"`, `native: true`, and `color: "#2563EB"`.
- Keep Code as the default agent; Data is only selected explicitly or through existing `default_agent` configuration.
- Clone Code's effective default permission rules so Data can use configured notebook/query tools and can still perform ordinary operations when explicitly needed. Notebook-only behavior is prompt guidance, not a permission boundary.
- Keep the prompt generic. Do not mention dbt, Snowflake, `vscode-notebook`, other MCP server names, or kernel setup.
- Do not copy `.kilo/rules`, skills, commands, or other bundle files from `/Users/imanol/code/kilocode-dbt`.

## Implementation

1. Add `packages/opencode/src/kilocode/agent/prompt/data.txt` with a concise notebook-focused role:
   - Use an active Jupyter notebook as the working surface.
   - If none is active, create a uniquely named, descriptive `<topic>.ipynb` in the workspace root without overwriting an existing file.
   - For every user request, append at least one focused code cell and execute it.
   - Preserve history: do not modify or delete existing cells unless explicitly asked; append diagnostic or corrected cells after failures.
   - Keep substantive data work and evidence in the notebook, and avoid changing non-notebook files unless explicitly requested or necessary.
   - Inspect cell output before answering, keep outputs and final summaries concise, and never claim execution when a notebook cell did not run.
   - Do not add notebook-tool preference, fallback, or kernel-selection instructions yet.

2. Register Data in `packages/opencode/src/kilocode/agent/index.ts`:
   - Import the Kilo-owned prompt.
   - Add Data in `patchAgents()` after Code has been created.
   - Give it a generic description such as “Run notebook-first data analysis by appending and executing cells for each request.”
   - Clone `agents.code.permission` rather than duplicating the permission policy or sharing a mutable array reference.
   - Leave `model`, `steps`, and other runtime tuning unset.
   - Keep all substantive registration logic in this Kilo-owned file; do not expand the hook in `packages/opencode/src/agent/agent.ts`.

3. Add `data` as a known built-in agent property in the `agent` schema in `packages/opencode/src/config/config.ts` with a narrow `kilocode_change` annotation. The existing record fallback still supports arbitrary custom names, while this explicit property gives generated schemas and SDK types first-class completion for `agent.data`.

4. Add Kilo-owned regression coverage in `packages/opencode/test/kilocode/data-agent.test.ts`:
   - Data appears in the default agent list and is visible, primary, and native.
   - It has the expected blue color, description, and notebook prompt.
   - It has no pinned model or step limit.
   - Representative permissions match the Code baseline, including normal edit capability, shell approval behavior, skills, and task/todo access.
   - Code remains the default agent.
   - Assert the prompt's essential contract—create/reuse a notebook, append a cell for each request, execute it, and preserve prior cells—without snapshotting the whole prose.

5. Update `packages/opencode/test/kilocode/agent-skill-permissions.test.ts` so `data` is included among non-system native agents that can use skills.

6. Run `./script/generate.ts` from the repository root after the schema change. Commit the resulting expected generated changes, including `packages/sdk/openapi.json` and `packages/sdk/js/src/v2/gen/types.gen.ts` where Data becomes a known agent configuration property. Inspect generated output and avoid unrelated churn.

7. Add `.changeset/notebook-first-data-agent.md` as a minor changeset for both `@kilocode/cli` and `kilo-code`, with a short user-facing description such as “Add a notebook-first Data agent that appends and runs a new notebook cell for each request.”

## Validation

Run the smallest relevant checks:

1. From `packages/opencode/`: `bun test ./test/kilocode/data-agent.test.ts ./test/kilocode/agent-skill-permissions.test.ts`
2. From `packages/opencode/`: `bun run typecheck`
3. From the repository root: `./script/generate.ts`
4. Re-run the targeted tests/typecheck if generation changes compiled SDK inputs.
5. From the repository root: `bun run script/check-opencode-annotations.ts`
6. Review the final diff to confirm shared upstream files contain only the narrow schema entry and existing registration hook remains unchanged.

## Explicitly out of scope

- Permission-enforced notebook-only operation
- Notebook MCP selection or fallback behavior
- Kernel discovery or selection
- dbt, Snowflake, or organization-context assumptions
- Bundled skills, rules, commands, or a Data subagent
- A fixed model or iteration limit
- Dedicated UI icons or client-specific hardcoded agent lists
