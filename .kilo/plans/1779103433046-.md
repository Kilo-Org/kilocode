# Plan: How to Write Good Specs — A Field Guide

## Objective
Produce a clean markdown reference document, approximately 1,000-2,000 words and under five printed pages, titled **"How to Write Good Specs — A Field Guide."** The document will be practical, workflow-oriented, and suitable as an internal shared standard for engineers and AI agents.

## Research

This section captures the source material gathered in the planning phase so the writing step does not need to re-research. Findings come from `task` agents and from direct fetches of the primary sources.

- Amazon Working Backwards / PR-FAQ — "An insider look at Amazon's culture and processes" (Bryar & Carr excerpt on aboutamazon.com), `workingbackwards.com` PR/FAQ template, Commoncog write-up. Fetched the Amazon excerpt directly.

### Joel Spolsky — what to keep

- A spec's most important function is to **design the program in prose** before writing code. Without one is "the single biggest unnecessary risk."
- Distinguish **functional spec** (user-visible behavior) from **technical spec** (implementation). The field guide is mainly about the functional/behavioral kind.
- Recommended ingredients of a good spec: disclaimer, single named author/owner, scenarios, non-goals, overview, exhaustive details (especially error cases), flagged open issues, side notes for testers/marketing/docs.
- Specs must stay **alive**: updated as decisions change, frozen only at code-complete.
- Tips: be readable, write simply, use screenshots/mockups, reread, and avoid heavy templates — "templates considered harmful."
- "Specs are like flossing: everybody knows they should be writing them, but nobody does." Useful tone-setter.

### RFC 2119 + RFC 8174 — normative language

- Defines `MUST` / `MUST NOT` (absolute), `SHOULD` / `SHOULD NOT` (recommended; exceptions must be understood and weighed), `MAY` (truly optional).
- RFC 8174 clarifies the special meaning applies **only when the words appear in ALL CAPS**; lowercase "must" or "should" carry their normal English meaning.
- Imperatives "MUST be used with care and sparingly… only where it is actually required for interoperation or to limit behavior which has potential for causing harm." This is the anti-overuse rule the field guide should cite.
- `SHOULD` is not "weak MUST"; it explicitly admits exceptions but requires the implementer to understand the consequences.

### Amazon Working Backwards / PR-FAQ — what to keep

- Start from the customer experience and work backwards to implementation.
- The PR (under one page) describes the desired customer experience; the FAQ (≤5 pages) pressure-tests feasibility, economics, ops, risk, and reasons for failure.
- "The goal isn't to explain all the excellent work you have done but rather to share the **distilled thinking** that has come from that work."
- Brevity is a forcing function. 10+ drafts and many leadership reviews are normal.
- Most PR/FAQs do **not** become products — "a feature, not a bug." Deciding **not** to build is a legitimate output.
- The PR/FAQ is a **living document** that continues to evolve after approval.

### Agent-oriented sources — what to keep

- An agent is "instructions (what it should do), guardrails (what it should not do), and tools (what it can do)."
- For agents, instructions must be **explicit**: clear actions, edge cases, exit conditions, structured outputs, and failure handling.
- Anthropic emphasizes: maintain simplicity, prioritize transparency about planning, and craft the agent-computer interface (tool docs and parameter naming) carefully — "think of it as writing a great docstring for a junior developer."
- Anthropic's harness work shows the value of bounded environments: an `init.sh`, a structured progress file, a feature list with explicit pass/fail markers, and instructions like "It is unacceptable to remove or edit tests." These are concrete patterns of agent-targeted specs.
- OpenAI's guide stresses: define the smallest agent that can own a clear task, define exit conditions for the run loop, and use **structured outputs** (typed schemas) to make completion verifiable.

### Distilled cross-source principles

- A spec exists to **enable a decision** and reduce ambiguity, not to document after the fact.
- Separate three layers: outcomes/goals, externally observable behavior, implementation choices.
- Make boundaries explicit: non-goals and out-of-scope items are part of the spec.
- Prefer concrete scenarios over abstract promises.
- Convert ambiguity into named decisions or flagged open issues.
- Use normative language deliberately and sparingly.
- Keep rationale close to but distinct from guidance.
- Optimize for readability and revisability; an unread spec aligns nothing.

### Pitfalls catalogued during research (with weak/better pairs)

- Implementation in place of behavior: "Use Redis for throttling" → "Reject more than 100 reset requests per account per hour."
- Vague adjectives: "Search should be fast" → "p95 search latency ≤ 300 ms at the baseline workload."
- Overusing `MUST`: applying it to cosmetic preferences instead of contract/safety requirements.
- Hidden ambiguity: "Authentication TBD" → "Open issue: signed cookies vs OAuth tokens, blocking before implementation because it affects logout and mobile."
- Missing non-goals: "Support import" → "CSV only this release; XLSX and scheduled imports are explicitly out of scope."
- Happy-path-only writing: define invalid inputs, duplicate submissions, expired sessions, partial failures, and retry behavior.
- Heavy templates: 25 mandatory sections regardless of task size; people stop using them.
- Frozen-at-kickoff specs that diverge silently from the shipping product.

### Spec-vs-other-artifact taxonomy (synthesis)

- `PRD` answers **why** and for whom; defines outcomes and success metrics.
- `Software / behavioral spec` answers **what exactly** the system does, including edge cases.
- `Design / technical spec` answers **how** the system should be built — architecture, interfaces, alternatives, trade-offs.
- `Implementation plan` answers **in what sequence** to build, verify, and roll out.
- `Agent-oriented spec` answers **how an AI agent should execute** a bounded task with explicit inputs, outputs, permissions, and stop conditions.

### Humans vs AI-agent readers (synthesis)

| Dimension | Human readers | AI-agent readers |
|---|---|---|
| Ambiguity tolerance | Moderate; can ask follow-ups | Low; ambiguity causes drift |
| Rationale | Helps buy-in and judgement | Useful but secondary |
| Constraints | Often implicit | Must be explicit |
| Examples | Helpful | Often essential |
| Completion criteria | May be informal | Must be testable |
| Permissions | Socially understood | Must be stated mechanically |
| Failure handling | Deferred to judgement | Specified up front |

Mirrored example:
- Human: "Prefer the smaller architecture change unless it materially increases operational risk."
- Agent: "Do not introduce a new service. Modify only the existing worker and API layer unless the current design cannot satisfy `R3`; if blocked, stop and report why."

## Source-informed framing
Ground the guide in a small set of recognizable, real-world references without turning it into a literature review:
- Joel Spolsky's functional spec guidance: specs as design-in-prose, scenarios, non-goals, explicit decisions, open issues, living documents.
- RFC 2119 + RFC 8174: precise use of `MUST`, `SHOULD`, `MAY`, and the warning not to overuse normative keywords.
- Amazon Working Backwards / PR-FAQ: start from customer value, iterate in writing, keep documents concise, use writing to expose weak ideas before implementation.
- Agent-oriented guidance from current AI practice (Anthropic, OpenAI Agents SDK): make inputs, outputs, permissions, stop conditions, and verification explicit when the reader is an AI agent.

## Proposed document structure
1. **Opening thesis: what a spec is for**
   - Define a software spec as a decision and alignment artifact that states expected behavior clearly enough to build, review, and verify.
   - State what it is not: not a PRD, not a design doc, not an implementation plan, not a task checklist.

2. **How specs fit with adjacent artifacts**
   - Use a compact comparison list or table:
     - PRD = why / user problem / outcomes
     - Software spec = what exactly must happen
     - Design doc = how the system should work
     - Plan = execution order and delivery steps
   - Mention that one project may need only one artifact, while higher-risk work may need several.

3. **Three or more spec types and when to use them**
   - Cover at least these three:
     - Behavioral or feature spec
     - Technical/interface spec
     - Agent or execution spec
   - Optionally mention product-intent specs if it helps clarify boundaries.
   - For each type, provide: purpose, when to use it, and what it should emphasize.

4. **What makes a spec good**
   - Present at least 5 concrete criteria; target 7-9 for completeness.
   - Candidate criteria:
     - Clear problem and purpose
     - Explicit scope and non-goals
     - Concrete scenarios and examples
     - Testable requirements and acceptance signals
     - Edge cases and failure modes included
     - Decisions and open questions separated
     - Appropriate precision in normative language
     - Traceability to verification / implementation readiness
     - Readability and brevity

5. **Common pitfalls, with examples**
   - Include several paired weak/better examples, such as:
     - Vague requirement: "fast" vs measurable latency target
     - Implementation leakage: "use Redis" vs required throttling behavior
     - Hidden ambiguity: "authentication TBD" vs explicit unresolved decision
     - Missing non-goals: "support import" vs limited first-release scope
     - Happy-path-only writing vs defined error and retry behavior
     - Overusing `MUST` where a preference is intended
   - Keep examples short and concrete.

6. **Humans vs AI agents**
   - Explain that humans tolerate some implicit context and can negotiate ambiguity; AI agents need operational specificity.
   - Contrast the two across a few dimensions: ambiguity tolerance, permissions, outputs, stop conditions, examples, verification.
   - Include 1-2 mirrored examples showing how a human-facing instruction becomes more explicit for an agent-facing spec.

7. **A universal spec structure**
   - Propose a simple reusable template that works for most cases:
     1. Title / status / owner
     2. Context and problem
     3. Goals and success criteria
     4. Scope and non-goals
     5. Actors, scenarios, and workflows
     6. Requirements and constraints
     7. Proposed behavior or operating contract
     8. Verification / acceptance
     9. Risks, dependencies, and open questions
   - Also provide a compressed 7-part version for smaller work if space allows.

8. **Closing: workflow guidance**
   - End with a short practical summary:
     - Write the spec early enough to affect decisions.
     - Revise it as understanding changes.
     - Use it to align implementation, review, testing, and delegation.

## Tone and formatting decisions
- Use concise prose with short sections and bullets for scanability.
- Avoid academic framing and avoid tooling-specific recommendations.
- Keep the total length near 1,400-1,700 words to satisfy the page limit comfortably.
- Use markdown tables only if they improve compression; otherwise prefer bullets to stay readable.
- Include a brief references note or inline source mentions, but avoid formal citation overhead unless it improves credibility.

## Execution checklist for the actual writing step
- Verify every user requirement is explicitly satisfied:
  - definition and distinctions
  - 3+ spec types
  - 5+ evaluation criteria
  - pitfalls with examples
  - humans vs AI agents
  - universal structure
  - clean markdown deliverable
- Ensure the final prose is practical and not overly abstract.
- Check that any use of `MUST` / `SHOULD` / `MAY` is explained consistently with RFC guidance.
- Keep examples domain-neutral and useful to working engineers.
- End with a polished markdown document only, ready to save as-is.
