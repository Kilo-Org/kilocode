# Plan: Add a `/stats` slash command to the Kilo CLI TUI

## Goal

Add a `/stats` TUI slash command that mirrors the `kilo stats` CLI command's output, scoped to the **current project, all-time**. It reuses the existing `aggregateSessionStats` Effect logic (already exported with a `kilocode_change` marker in `packages/opencode/src/cli/cmd/stats.ts:88`) and surfaces the result in an interactive TUI dialog via the established Kilo server-endpoint + SDK pattern.

## Background / decisions

- The TUI slash commands that *execute logic and display results* are "action/palette" commands registered via `useBindings(() => ({ commands: [...] }))` in the TUI. The Kilo-specific ones live in `packages/opencode/src/kilocode/kilo-commands.tsx` (`/profile`, `/teams`, `/remote`, `/kiloclaw`, `/indexing`). `/stats` belongs here.
- The aggregation logic in `aggregateSessionStats` runs server-side (it depends on `Session.Service` + `InstanceRef` + `Project.Info`), so it must be invoked over HTTP. The established Kilo pattern is a dedicated Kilo HttpApi group (`packages/opencode/src/kilocode/server/httpapi/groups/`) + handler (`.../handlers/`) registered in `server.ts` and added to `InstanceHttpApi` in the shared `api.ts` (inside the existing `kilocode_change` Kilo block, so no new marker logic needed — just append a line).
- All new code goes under `packages/opencode/src/kilocode/`, so **no `kilocode_change` markers are required** for the new files (per AGENTS.md). The only shared file touched is `packages/opencode/src/server/routes/instance/httpapi/api.ts`, where we append to the existing Kilo HttpApi block — that block is already bracketed by `kilocode_change start/end` markers, so we only add the import + `.addHttpApi(StatsApi)` line within it.
- User confirmed: mirror `kilo stats` output, scope = current project + all-time, reuse the stats logic the same way `kilo stats` does (server-side `aggregateSessionStats`).

- `StatsApi = HttpApi.make("stats").add(HttpApiGroup.make("stats").add(HttpApiEndpoint.get("stats", StatsPaths.stats, { query: WorkspaceRoutingQuery, success: described(SessionStats, "Aggregated session usage statistics") }).annotateMerge(OpenApi.annotations({ identifier: "stats.stats", summary: "Get session usage statistics", description: "Aggregate token usage, cost, model usage, and tool usage across sessions, scoped to the current project." }))).annotateMerge(OpenApi.annotations({ title: "stats", description: "Kilo session statistics routes." })).middleware(InstanceContextMiddleware).middleware(WorkspaceRoutingMiddleware).middleware(Authorization)).annotateMerge(OpenApi.annotations({ title: "kilo HttpApi", version: "0.0.1", description: "Kilo HttpApi surface." }))`.

### 2. New handler

**New file:** `packages/opencode/src/kilocode/server/httpapi/handlers/stats.ts`

Mirror `handlers/remote.ts`:
```ts
export const statsHandlers = HttpApiBuilder.group(InstanceHttpApi, "stats", (handlers) =>
  Effect.gen(function* () {
    const stats = Effect.fn("StatsHttpApi.stats")(function* () {
      const ref = yield* InstanceRef  // per-request context (current project)
      const project = ref?.project
      return yield* aggregateSessionStats(undefined /* days = all time */, "", project)
    })
    return handlers.handle("stats", stats)
  }),
)
```
- Import `aggregateSessionStats` from `@/cli/cmd/stats` and `InstanceRef` from `@/effect/instance-ref`.
- The `kilo stats` CLI handler (`stats.ts:69-80`) calls `aggregateSessionStats(args.days, args.project, ctx.project)` where `ctx` comes from `yield* InstanceRef`. We replicate that exactly with `days=undefined` (all-time) and `projectFilter=""` (current project). `aggregateSessionStats` throws if `projectFilter===""` and `currentProject` is null; guard by returning an empty/zero stats object if `!project` (or let it error and map to `HttpApiError.BadRequest`). Prefer: if `!project`, return a zero-stats object directly to avoid a 500.
- The handler group uses `InstanceRef` which is request-derived context (matches the HttpApi AGENTS guidance — `InstanceRef` is explicitly called out as request-derived, available via the middleware `InstanceContextMiddleware`).

### 6. TUI slash command registration

```ts
{
  name: "kilo.stats",
  title: "Stats",
  desc: "Show token usage and cost statistics for this project",
  category: "Kilo",
  slashName: "stats",
  run: async () => {
    try {
      const response = await sdk.client.stats.stats()
      if (response.error || !response.data) {
        dialog.replace(() => <DialogAlert title="Error" message="Failed to fetch session statistics." />)
        return
      }
      dialog.replace(() => <DialogStats stats={response.data} />)

      dialog.replace(() => <DialogAlert title="Error" message={`Failed to fetch stats: ${error}`} />)
    }
  },
},
```

## Risks / notes

- `aggregateSessionStats` reads **all** sessions from the DB then filters in memory (`getAllSessions` at `stats.ts:83`). For projects with many sessions this can be slow; the CLI prints a "Large dataset detected" notice at >1000 sessions. The TUI dialog should account for this latency — consider showing a loading state in the dialog, or a toast "Computing stats..." before the fetch. For the first implementation, the simple `await sdk.client.stats.stats()` with an error fallback is acceptable (matches `/profile`), but a follow-up could add a spinner. Note this in the plan but keep v1 simple.
- `InstanceRef` availability in the handler: the `InstanceContextMiddleware` provides request-derived `InstanceRef` context (confirmed by the HttpApi AGENTS.md middleware guidance). The `kilo stats` CLI uses `yield* InstanceRef` directly, so the handler can do the same within the request Effect.
- The `kilo stats` command already exists as a yargs subcommand; this plan adds the **TUI slash command** `/stats` only. No conflict — they are different surfaces (CLI vs TUI palette). The `slashName: "stats"` maps to `/stats` in the prompt input.
