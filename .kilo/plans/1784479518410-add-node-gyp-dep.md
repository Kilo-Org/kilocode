# Add `node-gyp` as a workspace dep so `bun install` provides it

## Goal

Make `node-gyp` available on `PATH` (and as a callable binary) immediately after `bun install` runs in this monorepo, so transitive native deps that invoke `node-gyp` during their install / build steps don't fail with "node-gyp not found".

Out of scope: the subsequent `zig` requirement raised by the user. To be handled in a follow-up plan.

## Context

- Repo is a Bun workspace monorepo (`packageManager: "bun@1.3.14"` in root `package.json`).
- Today, neither root nor any `packages/*/package.json` declares `node-gyp`.
- Native build paths that may invoke `node-gyp`:
  - `bun-pty` (`packages/opencode/package.json:133`, `packages/core/package.json:119`)
  - `@lydell/node-pty` (cataloged; used via `fix-node-pty` postinstall)
  - Anything else pulling in `node-gyp` transitively from `ai-sdk`, `effect`, etc.
- The failing step happens **after** `bun install` (i.e. during a subsequent build), so the user's request is to make the binary exist so the build can find it.

## Decision

Add `node-gyp` to **root `package.json` `devDependencies`** with range `^11.0.0`.

Rationale:
- Root devDeps are hoisted into `node_modules/.bin` and available to every workspace, which matches "any package may need native builds".
- v11 is the current major and supports Node 18+.
- `devDependencies` (not `dependencies`) is the correct class for a build tool.

## Changes

1. Edit `package.json` (repo root):
   - In `devDependencies`, add a new entry (alphabetically placed near `oxlint` / `prettier`):
     ```json
     "node-gyp": "^11.0.0",
     ```
   - Do **not** add it to `dependencies`, `overrides`, `trustedDependencies`, `catalog`, or `patchedDependencies`.

2. Refresh the lockfile so the binary is materialised:
   - Run `bun install` from repo root.
   - Verify `node_modules/.bin/node-gyp` (or `node_modules/node-gyp/bin/node-gyp.js`) is present.
   - The `bun.lock` file will be updated; commit it alongside `package.json`.

## What this does NOT do

- It does not auto-run `node-gyp` for any specific dep. If a dep needs a native build, that dep's own `install` / `prebuild` script is responsible for invoking it. The fix here is only that the binary now exists.
- It does not install a C/C++ toolchain (`gcc`/`clang`/`python`) nor a Zig toolchain. Those are separate environment prerequisites.
- It does not change `bunfig.toml`, build scripts, or any package's `postinstall`.

## Validation

After the change:
1. `bun install` from repo root completes without errors.
2. `ls node_modules/.bin/node-gyp` exists (or `bunx node-gyp --version` returns a version).
3. The user's original failing build (whatever they ran after `bun install`) gets past the "node-gyp not found" stage and instead fails at the next missing prerequisite (zig), confirming node-gyp is no longer the blocker.
4. `bun run lint` and `bun run typecheck` from repo root still pass (sanity-check that the dep addition didn't disturb workspace resolution).
5. (Optional) From `packages/opencode/`, run `bun run typecheck` to confirm the workspace still resolves.

## Risks

- `node-gyp` v11 requires Node 18+ to run; the repo already requires modern Node via `@types/node: 24.x`, so this is satisfied.
- If a workspace later wants to pin a different range, the root entry can be moved into `workspaces.catalog` — not doing that now to keep the diff minimal.

## Follow-up (out of scope here)

Plan separately for the `zig` dependency surfaced by the build pipeline. Likely candidates to investigate: `packages/opencode/script/kilocode/bubblewrap.ts:105` (`process.env.ZIG ?? "zig"`) and `src/lsp/server.ts:589` (`extensions: [".zig"]`).
