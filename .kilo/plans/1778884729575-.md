# Automated Doc Update PRs for Upstream Merges

## Problem Statement

After merging upstream opencode releases, Kilo documentation often becomes outdated. The goal is to automate the creation of PRs that update documentation to reflect upstream changes.

## Current State Analysis

### Existing Infrastructure

| Component | Purpose |
|---|---|
| `watch-opencode-releases.yml` | Polls upstream releases every 30 min, fires webhook on new release |
| `auto-docs.yml` | Triggers on merged `feat:` PRs, sends metadata to Val Town webhook |
| `script/upstream/merge.ts` | Transforms and merges upstream into Kilo, writes `.opencode-version` |
| `generate.yml` | Runs `./script/generate.ts` on push to `dev` branch |
| Val Town webhook (`DOC_WEBHOOK_URL`) | External service for auto-docs processing |
| External sync webhook (`OPENCODE_WATCH_SYNC_URL`) | Triggers upstream merge automation |

### Gap

The `auto-docs.yml` workflow triggers on **feature PR merges** but not on **upstream merge PRs**. When upstream is merged, the webhook is sent to the sync system, but no doc analysis/update PR is automatically created.

## Recommended Approach

### Primary Trigger: Upstream Merge to Main

Create a new workflow that triggers when an upstream merge PR is merged to `main`. This is the most precise trigger because:

1. **Directly correlated** to when code actually changes
2. **Leverages existing infrastructure** (upstream merge already creates a PR)
3. **Immediate feedback** — doc updates proposed right after merge lands
4. **Scoped to actual changes** — only analyzes docs that were affected by the merge

### Secondary Trigger: New Upstream Release (Cumulative Check)

Extend the existing `watch-opencode-releases.yml` to also trigger a doc analysis when a new upstream release is detected. This serves as a safety net:

1. **Catches cumulative drift** — if multiple releases passed without merges
2. **Independent of merge timing** — works even if merges are delayed
3. **Batch processing** — analyzes all changes since last known version

## Implementation Design

### Architecture Overview

```
┌─────────────────────────────────────────┐
│  watch-opencode-releases.yml            │
│  (every 30 min)                         │
│  Detects new upstream release           │
└────┬────────────────────────────────────┘
     │
     ├─► OPENCODE_WATCH_SYNC_URL (existing)
     │   Triggers upstream merge automation
     │
     └─► DOC_WEBHOOK_URL (extended)
         Triggers cumulative doc analysis
         
┌─────────────────────────────────────────┐
│  upstream-merge-docs.yml (NEW)          │
│  Trigger: pull_request_target closed    │
│  Condition: upstream merge PR merged    │
└────┬────────────────────────────────────┘
     │
     ├─► Calculate diff between upstream docs and current docs
     ├─► Spawn Kilo agent to analyze and propose updates
     └─► Create PR with proposed doc changes
```

### New Workflow: `upstream-merge-docs.yml`

**Trigger:** `pull_request_target` on `closed` when PR is merged and title matches upstream merge pattern (e.g., `chore: merge upstream opencode v*` or contains the upstream version tag).

**Steps:**

1. **Detect upstream merge PR**
   - Check if PR title matches upstream merge pattern
   - Extract upstream version tag from PR title or `.opencode-version` diff
   - Calculate the previous version from before the merge

2. **Compute doc diff**
   - Fetch upstream opencode repo at the new version tag
   - Identify all documentation files that changed between previous and new version
   - Filter to only files that have corresponding Kilo docs (not Kilo-specific docs)
   - Generate a structured diff report

3. **Spawn Kilo agent for analysis**
   - Use the Kilo CLI in headless mode to analyze the diff
   - Provide context about what changed and what Kilo-specific docs exist
   - Generate proposed updates that:
     - Sync upstream documentation changes
     - Preserve Kilo-specific branding and customizations
     - Update references to match Kilo terminology

4. **Create PR**
   - Branch: `kilo-bot/doc-sync-upstream-<version>`
   - Commit message: `docs: sync documentation with upstream opencode <version>`
   - PR title: `docs: sync documentation with upstream opencode <version>`
   - PR body: Summary of what changed, link to upstream release notes

### Extend Existing Workflow: `watch-opencode-releases.yml`

Add a step that triggers the doc webhook when a new release is detected, but with a different payload indicating it's a cumulative check:

```yaml
- if: steps.fetch.outputs.tag != '' && steps.cache.outputs.cache-hit != 'true'
  name: Trigger doc sync analysis
  env:
    URL: ${{ secrets.DOC_WEBHOOK_URL }}
    SECRET: ${{ secrets.DOC_WEBHOOK_SECRET }}
  run: |
    curl -sf -X POST "$URL" \
      -H "x-doc-sync-secret: $SECRET" \
      -H "Content-Type: application/json" \
      -d "{\"event\": \"upstream_release\", \"tag\": \"${{ steps.fetch.outputs.tag }}\"}"
```

## Key Decisions

### 1. What scope of docs to analyze?

**Recommendation:** Use the diff between the last merged upstream version and the new version. This is more precise than "last N PRs" because:

- Upstream merges are the actual boundary where Kilo's docs diverge
- Version tags provide clean before/after comparison points
- Avoids noise from intermediate upstream PRs that don't affect docs

### 2. How to identify upstream merge PRs?

**Recommendation:** Match on PR title patterns used by the upstream merge script. The merge script creates branches like `<author>/kilo-opencode-<version>`. We can:

- Match PR titles containing `opencode` and a version tag
- Or check for changes to `.opencode-version` in the PR diff
- Or add a specific label to upstream merge PRs (e.g., `upstream-merge`)

### 3. Kilo agent vs scripted approach?

**Recommendation:** Use Kilo CLI in headless mode for the actual doc analysis because:

- Already available as a binary
- Has context about the codebase
- Can reason about Kilo-specific vs upstream docs
- Can follow the project's style guide and conventions

However, the diff calculation and PR creation can be scripted for reliability.

### 4. Error handling and false positives?

**Recommendation:**
- Skip if no docs were changed in the upstream merge
- Use `workflow_dispatch` for manual re-runs
- Add a dry-run mode that just logs what would change
- Create PR as draft if confidence is low

## Alternative Approaches Considered

### A. Analyze last N upstream PRs

**Pros:** Catches more granular changes, can batch multiple PRs
**Cons:** Complex to track which PRs have been processed, noisy for non-doc PRs, harder to determine "last processed" boundary

### B. Scheduled doc sync (weekly/daily)

**Pros:** Simple to implement, catches everything
**Cons:** Not tied to actual changes, wastes resources, slower feedback

### C. Manual trigger only

**Pros:** Full control, no false positives
**Cons:** Easy to forget, defeats the purpose of automation

### D. Extend auto-docs.yml to also trigger on upstream merges

**Pros:** Reuses existing infrastructure
**Cons:** auto-docs.yml is designed for feature PRs with specific title patterns, upstream merges have different semantics, would complicate the workflow

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Kilo agent produces incorrect docs | Human review required, PR as draft initially |
| False positives on non-doc upstream merges | Skip if no docs changed |
| Webhook failures | Retry logic, manual trigger fallback |
| Rate limiting on GitHub API | Batch operations, use appropriate tokens |
| Upstream docs don't map cleanly to Kilo docs | Agent instructed to preserve Kilo-specific content |
