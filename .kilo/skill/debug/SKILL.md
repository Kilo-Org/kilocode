# Debug workflow

## Reproduce first

- Get a minimal, reliable reproduction before touching code.
- Run the failing command/script yourself to confirm the error is real and consistent.

## Read the stack bottom-up

- Start at the innermost frame (the actual crash site), not the top-level error message.
- Trace types/variables from the crash site outward. Do NOT guess — read the source.

## Bisect

- Binary search through commits or config options to isolate the change that introduced the bug.
- `git bisect` when the bug spans multiple commits and reproduction is cheap.
- Comment out half the code path to narrow the locus when bisect is too slow.

## Confirm before fixing

- Propose the fix, then verify by reproducing the original error one more time.
- Only make the edit after you have confirmed the root cause.
- After fixing, re-run the reproduction to confirm it passes AND existing tests still pass.

## Windows notes

- `Process.spawn` at `packages/opencode/src/util/process.ts:68` enforces `windowsHide: true` — use it for any subprocess spawn to avoid CMD window flashes. Do NOT edit this file; it is shared with upstream and marked with `kilocode_change`.
- `bun test` on Windows ends with a known `EACCES` error from `test/preload.ts` (SQLite WAL handles). This is NOT a test failure — judge by assertion counts, not the trailing error.

## Subagents

- Use `failing-test-triage` for test failure diagnosis when the orchestrator delegates test-root-cause work to it. It is read-only and specialized for root-cause analysis.
