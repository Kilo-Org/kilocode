# Out-of-the-Box Blind-Spot Review: PR #11712

## Review scope

Read all six specialist reports first, then reviewed the full `main...HEAD` change set and relevant surrounding implementation with emphasis on accessibility, internationalization, licensing/assets, platform behavior, migrations/versioning, generated API consistency, endpoint semantics, cancellation/atomicity, packaging, test realism, multi-root/worktree behavior, offline operation, and failure recovery.

This review adds five findings not already stated in the specialist reports. Existing findings are referenced where a new issue compounds them rather than repeated.

## Findings

### OUTOFTHEBOX-001 — High — A process crash can permanently lose managed skills because transaction recovery exists only in memory

- **References:** `packages/opencode/src/kilocode/stack/runtime.ts:290`, `packages/opencode/src/kilocode/stack/runtime.ts:304`, `packages/opencode/src/kilocode/stack/runtime.ts:321`, `packages/opencode/src/kilocode/stack/runtime.ts:348`, `packages/opencode/src/kilocode/stack/runtime.ts:363`, `packages/opencode/src/kilocode/stack/runtime.ts:397`, `packages/opencode/src/kilocode/stack/store.ts:267`, `packages/opencode/src/kilocode/stack/store.ts:314`, `packages/opencode/src/kilocode/stack/service.ts:429`, `packages/opencode/src/kilocode/stack/service.ts:476`, `packages/opencode/src/kilocode/stack/service.ts:522`
- **Evidence / reproduction:** Skill replacement/removal first renames the live skill into a randomly named `.stack-backup-*` directory and tracks that move only in the transaction closure's `backups` map. Configuration rollback similarly depends on scoped temporary files and in-memory `WeakMap` entries. Effect finalizers handle ordinary failures and interruption, but cannot run after `SIGKILL`, host crash, power loss, or runtime termination. Kill the CLI after `runtime.Transaction.remove/install` renames a live skill (lines 321/348) but before config commit and `transaction.complete()`. On restart there is no journal scan or recovery path for `.stack-backup-*`; the skill remains absent from `.kilo/skills`, and the temporary backup may later be treated as an unrelated directory. Killing after config rename but before completion creates the converse risk: committed config/receipts and skill moves cannot be reconciled from durable intent.
- **Impact:** The operation is described and reviewed as transactional, but its atomicity is process-scoped rather than crash-safe. A crash during Apply can leave project files and receipts inconsistent, hide a previously working skill, and require manual forensic recovery. This materially worsens the failure-recovery implications of `PERFORMANCE-001`: aborting the client is safe only if the server survives and finishes; a backend restart or extension-host/backend termination during that abandoned apply can strand state.
- **Recommendation:** Add a durable transaction journal under `.kilo` containing transaction ID, original/destination paths, expected fingerprints, config revisions, and phase; fsync journal and staged data before destructive renames; atomically advance phases; and recover or roll back incomplete journals at Stack service startup before serving reads/applies. Prefer platform-supported atomic swap patterns where possible. Add subprocess tests that pause at each destructive phase, force-kill the process, restart it, and assert a deterministic restored or completed state with no orphan backups.

### OUTOFTHEBOX-002 — Medium — The read-only detection endpoint is modeled as cacheable GET despite scanning mutable workspace state

- **References:** `packages/opencode/src/kilocode/server/httpapi/groups/stack.ts:122`, `packages/opencode/src/kilocode/server/httpapi/groups/stack.ts:127`, `packages/opencode/src/kilocode/server/httpapi/handlers/stack.ts:65`, `packages/sdk/openapi.json:16375`, `packages/kilo-vscode/src/stack/client.ts:79`
- **Evidence / reproduction:** `/kilocode/stack/detect` is a `GET` whose response is derived from the current filesystem, and the endpoint supplies no `Cache-Control: no-store` or validator. Generated SDK clients therefore issue an ordinary cacheable GET. Put a caching reverse proxy/service worker in front of `kilo serve`, call detect, change `package.json` or project files without changing the URL/directory query, and call detect again: HTTP caches are permitted to reuse the first successful response. GET also makes an expensive recursive scan eligible for speculative/link tooling behavior in generic clients. This is distinct from the authorization and resource-bound concerns in `SECURITY-004` and `SECURITY-005`.
- **Impact:** Users can receive stale detections and unknowingly configure resources for technologies no longer present, while tests against direct localhost requests continue to pass. Deployment through a browser-facing console proxy, enterprise gateway, or future hosted transport makes this more likely.
- **Recommendation:** Make detect a `POST` because it initiates an explicit, potentially expensive computation, or return `Cache-Control: no-store, private` and a response revision tied to the scan inputs. Regenerate OpenAPI/SDK after changing semantics and add an HTTP-level test that modifies a manifest between calls and verifies the second response is fresh through the supported deployment path.

### OUTOFTHEBOX-003 — Medium — The console wizard is entirely English and bypasses the translation system

- **References:** `packages/kilo-console/src/routes/config/StackRoute.tsx:60`, `packages/kilo-console/src/routes/config/StackRoute.tsx:69`, `packages/kilo-console/src/routes/config/StackRoute.tsx:84`, `packages/kilo-console/src/routes/config/StackRoute.tsx:218`, `packages/kilo-console/src/routes/config/stack/StackRail.tsx:16`, `packages/kilo-console/src/routes/config/stack/StackResourcesStep.tsx:21`, `packages/kilo-console/src/routes/config/stack/StackResourcesStep.tsx:56`, `packages/kilo-console/src/routes/config/stack/StackResourcesStep.tsx:160`, `packages/kilo-console/src/routes/config/stack/StackResourcesStep.tsx:239`
- **Evidence / reproduction:** The console Stack route and all step components render literal English strings, including navigation accessible names, errors, transaction status, controls, platform/auth labels, and security-sensitive write/authentication guidance. There is no locale/translation hook in these components. Open Kilo Console under a non-English locale: unlike localized product surfaces, Project Stack remains English. The VS Code implementation does use `language.t(...)`, so the two clients already diverge.
- **Impact:** The new always-visible console route from `IMPACT-002` is not localizable, including the text users rely on before enabling executable MCP resources. Translation work in `packages/kilo-i18n` cannot affect this client, and future wording fixes must be duplicated manually across UIs.
- **Recommendation:** Route all console Stack copy through the console's localization mechanism (or establish one shared with `@kilocode/kilo-i18n`), use stable keys shared with the VS Code wizard where semantics match, and localize platform/auth/status values rather than concatenating English fragments. Add one non-English rendering test and a key-completeness check for both clients.

### OUTOFTHEBOX-004 — Medium — All non-English prompt-popover dictionaries deliberately ship English strings

- **References:** `packages/kilo-vscode/webview-ui/src/i18n/ar.ts:1652`, `packages/kilo-vscode/webview-ui/src/i18n/de.ts:1715`, `packages/kilo-vscode/webview-ui/src/i18n/es.ts:1705`, `packages/kilo-vscode/webview-ui/src/i18n/fr.ts:1728`, `packages/kilo-vscode/webview-ui/src/i18n/ja.ts:1675`, `packages/kilo-vscode/webview-ui/src/i18n/zh.ts:1615`, `packages/kilo-vscode/webview-ui/src/components/shared/StackPopover.tsx:65`, `packages/kilo-vscode/webview-ui/src/components/shared/StackPopover.tsx:94`, `packages/kilo-vscode/webview-ui/src/components/shared/StackPopover.tsx:100`
- **Evidence / reproduction:** The PR adds the same English values for the twelve `prompt.stack.*` and `settings.experimental.projectStack.*` keys to every one of the 19 non-English webview dictionaries. For example, French contains “No stack configured. Click the gear...” and Arabic contains the same left-to-right English sentences. Because keys are present, normal missing-key fallback/translation completeness tooling cannot identify them as untranslated. Select French, Arabic, Japanese, or Chinese and enable Project Stack; the prompt popover, tooltips/accessible labels, and setting description switch to English.
- **Impact:** This is a user-visible localization regression in the ordinary chat surface and an accessibility problem for screen-reader users because control names are untranslated. In RTL locales it also injects unmarked English text into Arabic layout. It compounds `IMPACT-003`: enabling a seemingly small localized prompt feature both starts network work and introduces untranslated controls.
- **Recommendation:** Remove copied English entries from locale dictionaries so the explicit fallback mechanism remains observable, or provide actual translations. Add an i18n lint that rejects values identical to English for newly added user-facing keys unless allowlisted (brand terms such as MCP may be exempt), and add locale snapshots for one RTL and one CJK language.

### OUTOFTHEBOX-005 — Medium — Third-party dashboard icon attribution is omitted from the shipped VS Code extension

- **References:** `packages/kilo-console/public/licenses/dashboard-icons.txt:1`, `packages/kilo-console/public/licenses/dashboard-icons.txt:189`, `packages/kilo-vscode/assets/icons/dashboard/apache-airflow.svg:1`, `packages/kilo-vscode/assets/icons/dashboard/w3c.svg:1`, `packages/kilo-vscode/src/StackPanelProvider.ts:279`, `packages/kilo-vscode/.vscodeignore:23`
- **Evidence / reproduction:** The same third-party dashboard icon set is copied into both console and extension asset trees. Only the console adds `public/licenses/dashboard-icons.txt`, identifying Apache-2.0/Homarr attribution; there is no corresponding license/NOTICE file under `packages/kilo-vscode`, and repository search finds no packaging step that transfers the console notice into the VSIX. Package the extension and inspect the VSIX: dashboard assets are included because the webview loads `assets/icons/dashboard`, but their dedicated attribution is absent. The license text itself notes redistribution/NOTICE obligations and identifies the upstream copyright holder.
- **Impact:** The VSIX redistributes third-party copyrighted/trademark-bearing assets without the attribution bundled for the console artifact, creating release/compliance risk. Duplication noted in `IMPACT-004` therefore has a legal/package-integration consequence beyond binary size.
- **Recommendation:** Add the authoritative third-party attribution/NOTICE to the VS Code package and verify it is included in the VSIX, ideally generate both product copies from one source-of-truth asset manifest with per-file source, license, copyright, and trademark review. Add a packaging test that every shipped third-party asset group has a shipped notice, and have legal/product verify whether individual vendor marks permit this use.

## Cross-review themes

- **Trust and exact authorization:** `SECURITY-002`, `SECURITY-003`, and `CODEQUALITY-001` combine into one release-blocking boundary: Marketplace provenance is unsigned, displayed trust is not bound to the installed publisher/item, and the reviewed plan hash does not bind exact item content. These should be fixed as one protocol, not independently patched UI checks.
- **Project identity and state isolation:** `CODEQUALITY-002`, `CODEQUALITY-003`/`GENERIC-003`, and `IMPACT-006` show that config source selection, async summary publication, and fixed persistence paths disagree about which project owns state. Multi-root and Agent Manager/worktree scenarios need end-to-end tests that switch directories during delayed load/preview/apply and verify both UI and disk targets.
- **Cancellation versus durability:** `PERFORMANCE-001` and `PERFORMANCE-002` correctly call for cancellation/coalescing, but cancellation must be paired with the crash-safe recovery in `OUTOFTHEBOX-001`; otherwise lifecycle cleanup can conceal an operation whose durable state is unknown.
- **Offline semantics:** `GENERIC-004` conflicts with UI blocking policy, while `IMPACT-003` causes unsolicited Marketplace access. A coherent offline contract should distinguish local state load, safe cleanup, catalog refresh, and install actions, and both clients should expose the same blocking conflict codes.
- **Release/package readiness:** `GENERIC-001`, `GENERIC-002`/`CODEQUALITY-005`, missing changesets, `OUTOFTHEBOX-004`, and `OUTOFTHEBOX-005` indicate the implementation is not release-ready even apart from runtime correctness: focused tests/guards fail, versioning is absent, localization is masked by copied English, and one shipped artifact lacks attribution.
- **Default exposure and cost:** `IMPACT-001` through `IMPACT-004` show that an experimental UI flag does not form a product boundary. Backend initialization, command/console exposure, networking, and artifact cost remain default; gating and lazy loading need to be applied consistently across server, SDK clients, extension, and console.

## Conclusion

Five new blind-spot findings were identified: one high-severity crash-atomicity/recovery defect and four medium-severity endpoint semantics, internationalization, and asset-licensing/package-integration issues. The prior reports' findings remain applicable and are not duplicated here.
