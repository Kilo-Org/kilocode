## Task 1: Create TUI Swap Command Module

**Action**: Create `packages/opencode/src/devilcode/workflow-tui/commands/team-swap.ts`

```typescript
import type { Command, RegisterFn } from "./types"
import type { PositionSwapResult } from "../../team/position-swap"

export type TeamSwapCommandHandlers = {
  swapPosition: (position: string, provider: string, model: string) => Promise<PositionSwapResult>
}

export function createSwapCommand(handlers: TeamSwapCommandHandlers): Command {
  return {
    id: "team-swap",
    name: "team swap",
    description: "Swap a team position's provider/model mid-workflow",
    aliases: ["swap"],
    keybind: undefined,
    hidden: false,
    enabled: () => true, // Validation happens in handler
    action: async (args: string) => {
      const parts = args.trim().split(/\s+/)
      if (parts.length < 3) {
        return {
          success: false,
          message: "Usage: team swap <position> <provider> <model>",
        }
      }

      const [position, provider, model] = parts
      const result = await handlers.swapPosition(position, provider, model)

      if (result.success) {
        return {
          success: true,
          message: `Swapped ${position}: ${result.previousProvider}/${result.previousModel} → ${result.newProvider}/${result.newModel}`,
        }
      } else {
        return {
          success: false,
          message: `Swap failed: ${result.error}`,
        }
      }
    },
  }
}

export function registerTeamSwapCommand(register: RegisterFn, handlers: TeamSwapCommandHandlers): () => void {
  return register(createSwapCommand(handlers))
}
```

# Team Orchestrator Guide

The Team Orchestrator is Devil Code's system for managing multi-agent teams that execute structured workflows. This guide covers creating teams, running workflows, and live team editing.

## Overview

A team in Devil Code consists of:
- **Positions**: Roles like Senior Developer, Reviewer, QA Tester
- **Providers**: AI model providers (Anthropic, OpenAI, Google, etc.)
- **Models**: Specific models assigned to each position
- **Capabilities**: What each position can do (code generation, review, etc.)

Teams execute through a 7-stage workflow: plan → challenge → contract → build → review → ship → retro.

## Creating a Team

### Using the TUI

1. Start Devil Code: `devil`
2. Run `/team init` to launch the team builder
3. Select a quickstart template or build from scratch
4. For each position:
   - Choose from the canonical library (11 positions)
   - Assign a provider and model
   - Set effort level (low/medium/high)
5. Save the team

### Using VS Code

1. Open the Agent Manager panel
2. Navigate to the "Team Builder" tab
3. Click "New Team" or load an existing team
4. Configure positions using the visual editor
5. Save to persist changes

## Starting a Workflow

```bash
# In TUI
/workflow start

# Or with a specific team
/workflow start --team my-custom-team
```

The workflow proceeds through stages, with each stage dispatching to the appropriate position based on capability matching.

## Live Team Editing

**New in Phase 10**: You can swap a position's provider and model mid-workflow without aborting execution.

### Command Syntax

```bash
/team swap <position> <provider> <model>
```

### Example

```bash
# Swap the senior-developer position to use Google's Gemini
/team swap senior-developer google gemini-2.5-pro
```

### What Happens

1. **In-flight tasks**: Any task currently executing continues on the old model. It will complete normally.
2. **New tasks**: Tasks dispatched after the swap use the new model.
3. **Concurrency**: If the new model has different concurrency limits, slots rebalance at the next wave boundary.

### When to Swap

- Model is underperforming on current tasks
- Cost optimization (swap to cheaper model for remaining work)
- Capability mismatch discovered mid-workflow
- Testing different models for comparison

### Limitations

- Cannot swap tier (worker → orchestrator requires editing team config)
- Cannot swap mid-task (only between task dispatches)
- Invalid provider/model errors surface at next task start

## Monitoring Workflows

### Stage Indicator

The TUI header shows the current stage and which position is executing:

```
[build] senior-developer (anthropic/claude-sonnet-4-20250514)
```

### Telemetry Dashboard

In VS Code, the Telemetry tab shows:
- Success rate per team
- Stall rate per position
- Cost per workflow
- Duration by stage

## Troubleshooting

### "Position not found"

The position name must match exactly. Use `/team list` to see available positions.

### "Delegation violation"

In hierarchical routing, the parent role must be allowed to delegate to the target position. Edit the team config to add the position to `canDelegate`.

### Tasks stalling

Check the telemetry dashboard for stall rates. Consider swapping to a different model or increasing effort level.

# Workflow TUI Migration Guide

This guide helps users migrate from the legacy workflow-tui (pre-Phase 5) to the new Team Orchestrator system.

## What Changed

### Team Configuration

| Old (Pre-Phase 5) | New (Team Orchestrator) |
|-------------------|------------------------|
| 5 hardcoded presets | 11-position canonical library |
| Role names: `reviewer`, `developer` | Position IDs: `reviewer`, `senior-developer`, `developer` |
| Flat config structure | Hierarchical with capabilities |
| No validation | Stage coverage validation |

### Commands

| Old Command | New Command | Notes |
|-------------|-------------|-------|
| `/team init` | `/team init` | Now launches interactive builder |
| `/team list` | `/team list` | Shows all saved teams |
| N/A | `/team swap <pos> <prov> <model>` | New: live editing |
| N/A | `/team export <path>` | New: JSON export |
| N/A | `/team import <path>` | New: JSON import |

### File Locations

| Old | New |
|-----|-----|
| In-memory only | `~/.local/share/kilo/teams/<id>.json` (user-level) |
| N/A | `.planning/team.json` (project-local override) |

## Migration Steps

### Automatic Migration

If you have a workflow running with an old-style team config, Devil Code automatically migrates it:

1. Old role names mapped to canonical positions
2. Missing capabilities inferred from role name
3. Stage coverage validated
4. Migrated config saved to new location

### Manual Migration

For custom team configs:

1. Export your current config (if possible)
2. Run `/team init` and select "Build from scratch"
3. Re-create positions using the canonical library
4. Assign providers and models
5. Save with a new name

### Preset Mapping

| Old Preset | New Equivalent |
|------------|----------------|
| Solo Enhanced | Solo Team (quickstart) |
| Code Review Pair | Review Pair (quickstart) |
| Full Stack Team | Full Stack (quickstart) |
| CI/CD Pipeline | CI/CD Team (quickstart) |
| Research Team | Research Team (quickstart) |

## Breaking Changes

### Removed Features

- **Direct role editing**: Roles are now positions with fixed capabilities. To change capabilities, edit the position in the team builder.
- **Inline config override**: Use project-local `.planning/team.json` instead.

### Changed Behavior

- **Stage assignment**: Stages now match positions by capability, not role name. A position must have the required capability to execute a stage.
- **Concurrency**: Now enforced per-position with `maxConcurrent` limit.
- **Delegation**: Hierarchical routing enforces `canDelegate` rules.

## Rollback

If you need to revert to pre-migration behavior:

1. The migration does not delete your old config
2. Contact support for assistance with complex rollbacks
3. Consider filing an issue if migration caused data loss

## FAQ

### Q: My workflow broke after migration

Check the stage coverage indicator. The new system requires at least one position per stage capability. Add missing positions or adjust capabilities.

### Q: Can I use my old team config JSON?

Not directly. The schema changed. Use `/team import` with a new-format JSON, or manually re-create the team.

### Q: Where did my presets go?

Presets are now "quickstart templates" loaded via `/team init`. Select one to populate your team.

### Q: How do I see what changed?

Run `/team list` to see all teams. Run `/team export <path>` to export current config for inspection.

## Getting Help

- [Team Orchestrator Guide](/collaborate/teams/team-orchestrator-guide) - Full feature documentation
- [GitHub Issues](https://github.com/9thLevelSoftware/kilocode/issues) - Report migration problems
