# General Review

## Findings

- [High] Event Service connection failures are swallowed, so clients can enter "ready/connected" without a WebSocket. `EventServiceClient.connect()` catches both auth and transient handshake failures, calls `handleAuthFailure()`/`scheduleReconnect()`, and then resolves instead of rejecting in `packages/kilo-vscode/src/kiloclaw/event-service-client.ts:99` and `packages/opencode/src/kilocode/claw/event-service-client.ts:98`. The VS Code provider's `WebSocketAuthError` handling is therefore unreachable at `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:381`, and the TUI sets `connected=true` after `connect()` returns at `packages/opencode/src/kilocode/claw/hooks.ts:211`. Verify with an invalid Event Service token or unreachable `/connect`: the UI may proceed as connected rather than showing the upgrade/auth/network failure path.

- [High] `/kilo/claw/chat-credentials` returns the stored Kilo credential itself as the chat bearer token. The route directly returns `auth.key` or `auth.access` as `token` in `packages/kilo-gateway/src/server/routes.ts:615`, with comments stating it is the long-lived Kilo JWT at `packages/kilo-gateway/src/server/routes.ts:590`. This expands the blast radius of any local backend/webview compromise from chat-scoped credentials to the user's broader Kilo credential. A human should verify whether kilo-chat/event-service can instead receive a short-lived, audience-scoped token.

- [High] Legacy API-key and migrated org credentials may be incompatible with the new kilo-chat/Event Service auth path. The route assumes both `auth.type === "api"` and `auth.type === "oauth"` tokens are bearer credentials accepted by services that "verify it directly with NEXTAUTH_SECRET" in `packages/kilo-gateway/src/server/routes.ts:617`, but legacy migration stores plain API keys for non-org users and org-scoped API keys inside an OAuth-shaped record at `packages/kilo-vscode/src/legacy-migration/migration-service.ts:513`. Those values are then sent as `Authorization: Bearer ...` by the chat clients in `packages/kilo-vscode/src/kiloclaw/kilo-chat-client.ts:224` and `packages/opencode/src/kilocode/claw/kilo-chat-client.ts:215`. Verify upgraded users with existing/migrated Kilo Gateway API keys do not get persistent 401/403 from chat.

- [High] TUI KiloClaw drops action approval blocks, making bot approval prompts unusable there. The protocol defines `ActionsBlock` in `packages/opencode/src/kilocode/claw/types.ts:49` and the HTTP client exposes `executeAction()` in `packages/opencode/src/kilocode/claw/kilo-chat-client.ts:151`, but message conversion keeps only text blocks in `packages/opencode/src/kilocode/claw/client.ts:73`. The VS Code webview renders actions, but the TUI shows only text/"Thinking..." and provides no approve/deny path. Verify with a bot response containing an `actions` block.

- [Medium] Re-opening/reloading the VS Code KiloClaw webview can create duplicate Event Service clients and handlers. A successful `kiloclaw.ready` run creates new `EventServiceClient`/`KiloChatClient` instances in `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:359` and appends event handlers at `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:510`, but `init()` does not disconnect or clear the previous successful chat stream before bootstrapping again at `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:250`. Cleanup only runs on attach/dispose at `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:1167`. Verify by reloading the webview after a successful init and checking for multiple `/connect` sockets or duplicated event side effects.

- [Medium] CLI chat tokens are never refreshed after the initial credential fetch. The token envelope includes `expiresAt` in `packages/opencode/src/kilocode/claw/types.ts:30`, but the TUI clients close over `input.envelope.token` forever in `packages/opencode/src/kilocode/claw/client.ts:138`, and no `onUnauthorized` handler is passed to either Event Service or chat HTTP client. Long-lived TUI sessions can fail after token expiry and keep retrying with the same expired bearer. Verify by using a short-lived token and waiting through expiry before sending or reconnecting.

- [Medium] VS Code unread state does not update when a conversation is marked read. The UI computes unread from `lastReadAt < lastActivityAt` in `packages/kilo-vscode/webview-ui/kiloclaw/components/ConversationList.tsx:108`, but `markRead()` only sends the HTTP request and does not update local conversation state in `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:1003`. The event map defines `conversation.read` at `packages/kilo-vscode/src/kiloclaw/types.ts:153`, but `attachEventHandlers()` does not handle it. Verify that selecting an unread conversation clears the unread indicator without requiring a full conversation refresh.

- [Medium] Approval resolution from another client will not update live in the VS Code panel. The event type `action.executed` is declared in `packages/kilo-vscode/src/kiloclaw/types.ts:156`, but the provider handles only `action.delivery_failed` around `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:641`. If another client resolves an approval, this panel can continue rendering active action buttons until a full message refresh. Verify by resolving the same approval from another client/session.

- [Medium] Rapid TUI conversation switches can display messages for the wrong active conversation. `selectConversation()` sets the active id, awaits history/status requests, and returns the fetched messages in `packages/opencode/src/kilocode/claw/client.ts:361`; the hook then unconditionally applies `result.messages` in `packages/opencode/src/kilocode/claw/hooks.ts:107`. With a slow network, selecting A then B can leave B active while A's later response overwrites the message list.

- [Low] VS Code message infinite-scroll can issue repeated identical history requests while the scrollbar remains near the top. `onScroll` posts `loadMoreMessages(activeId, oldest.id)` on every scroll event below 50px in `packages/kilo-vscode/webview-ui/kiloclaw/components/MessageArea.tsx:126`, and the provider has no in-flight guard or duplicate `before` suppression in `packages/kilo-vscode/src/kiloclaw/KiloClawProvider.ts:895`. This can amplify API traffic and create flicker under slow responses.
