Ask the user structured follow-up questions and receive answers as tool output.

## When to use

- Clarify ambiguous requirements that block progress
- Ask the user to choose between implementation options
- Gather preferences that materially affect code or documentation changes
- Confirm missing project-specific details before irreversible work
- Collect one or more selections in a structured format instead of free text

## When NOT to use

- Infer the answer from repository state or existing instructions — use `Read`, `Grep`, or `Glob`
- Record task progress — use `TodoWrite`
- Delegate research or implementation work — use `Task`

## Examples

- "Which package should own this feature?" → ask a single-choice architecture question
- "Which of these output formats do you want?" → present labeled options and mark one recommended
- "Which environments should this support?" → set `multiple: true` for multi-select answers
- "Need a custom answer option" → rely on the default typed response instead of adding `Other`

## Constraints

- Keep the header at 30 characters or fewer.
- Do not add an `Other` or catch-all option when `custom` is enabled.
- Set `multiple: true` only when more than one answer should be selectable.
- Put the recommended option first and suffix it with `(Recommended)` when you want to guide the choice.
