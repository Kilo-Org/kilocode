Run shell commands in `${shell}` on `${os}` with optional timeout and streamed output.
Use `${tmp}` for temporary work outside the workspace.

## When to use

- Run terminal-native workflows such as `git`, `npm`, `bun`, `docker`, or `gh`
- Execute package scripts, tests, builds, or CLIs that must run in a shell
- Inspect repository state with commands like `git status`, `git diff`, or `git log`
- Run multiple independent shell commands in parallel as separate tool calls
- Create commits or pull requests only when the user explicitly asks for them

## When NOT to use

- Read file contents or directory listings for analysis — use `Read` or `Glob`
- Search file contents — use `Grep`
- Edit or create files directly — use `Edit`, `Write`, or `ApplyPatch`
- Fetch or search web content — use `WebFetch` or `WebSearch`

## Examples

- "Run unit tests in `packages/opencode`" → set `workdir` instead of using `cd`
- "Check git status and git diff" → send separate `bash` calls in parallel
- "Run `mkdir` in path with spaces" → quote path with double quotes
- "Create a PR with `gh`" → inspect branch state first, then run `gh pr create`

## Constraints

- Use `workdir` instead of `cd <directory> && <command>`.
- Verify parent directories before commands that create files or directories.
- Quote every path containing spaces with double quotes.
- Prefer `Glob`, `Grep`, `Read`, `Edit`, and `Write` over shell `find`, `grep`, `cat`, `sed`, `awk`, or `echo` for codebase operations.
- Expect output beyond `${maxLines}` lines or `${maxBytes}` bytes to be truncated to a saved file.
- Write a 5-10 word `description` when possible.
- Chain dependent commands with shell-appropriate sequencing: ${chaining}
- Create commits or push branches only when the user explicitly requests it.
