# PR #11712 Non-Participant Impact Review

## Overview

PR #11712 adds Project Stack discovery and installation across the CLI backend, Kilo Console, and VS Code extension. This review examined `main...HEAD` with emphasis on users who do not enable or use Project Stack. The feature is partly opt-in in VS Code, but its backend, command registration, console code, assets, configuration schema, and shared styles are loaded or shipped on default paths.

Findings are listed below. Exact references point to the reviewed HEAD.

## Findings

### IMPACT-001 — High — The default CLI/server runtime eagerly initializes the entire Stack and Marketplace subsystem

- **References:** `packages/opencode/src/effect/app-runtime.ts:63`, `packages/opencode/src/effect/app-runtime.ts:126`, `packages/opencode/src/kilocode/stack/service.ts:186`, `packages/opencode/src/kilocode/stack/service.ts:189`, `packages/opencode/src/kilocode/stack/service.ts:563`, `packages/opencode/src/kilocode/stack/catalog/index.ts:296`, `packages/opencode/src/kilocode/stack/catalog/index.ts:306`, `packages/opencode/src/kilocode/stack/catalog/data.ts:8`, `packages/opencode/src/server/routes/instance/httpapi/server.ts:23`, `packages/opencode/src/server/routes/instance/httpapi/server.ts:207`
- **Evidence / reproduction:** Start any normal Kilo path that builds `AppRuntime` or the HTTP server, without opening Project Stack. `StackService.defaultLayer` is included in both default service graphs. Constructing it acquires `Marketplace.Service`, `StackStore.Service`, and `StackRuntime.Service`; importing the service also constructs and synchronously schema-validates the built-in 3,000-line catalog through `builtin = register(...)`. The VS Code setting only controls webview exposure and does not gate these backend layers.
- **User impact:** Every CLI, TUI, headless, and server user pays additional module parsing, catalog allocation/validation, service construction, and binary/package weight even when Project Stack is never used. A catalog validation defect can now fail ordinary startup. This also expands the default runtime dependency graph with filesystem, locking, archive, YAML, and marketplace code unrelated to normal sessions.
- **Recommendation:** Make Stack routes/services lazy or conditionally provided. Keep catalog construction behind the first `/stack/*` request, and avoid adding `StackService` to the process-wide `AppRuntime` unless a caller requests it. Add a startup benchmark and a smoke test proving regular CLI/server startup does not initialize Stack or Marketplace.

### IMPACT-002 — Medium — Project Stack is exposed by default despite the VS Code experimental flag being off

- **References:** `packages/kilo-vscode/package.json:127`, `packages/kilo-vscode/package.json:938`, `packages/kilo-vscode/src/extension.ts:268`, `packages/kilo-vscode/src/extension.ts:370`, `packages/kilo-console/src/routes/config/sections.tsx:78`, `packages/kilo-console/src/routes/config/sections.tsx:173`, `packages/kilo-console/src/index.tsx:27`
- **Evidence / reproduction:** Install the extension with a fresh profile where `kilo-code.new.experimental.projectStack` remains its default `false`. Open the Command Palette: `Kilo Code: Project Stack` is still registered with no `when` clause. Invoking it creates the panel and connects to the backend. In Kilo Console, every project settings page includes the `Project > Stack` navigation and route unconditionally; there is no corresponding feature flag.
- **User impact:** Users who do not opt into an experimental feature still see it in command search and project settings. Accidental invocation starts the backend if needed and triggers marketplace requests. This makes the experimental setting misleading because it hides only the prompt-area popover, not the feature itself.
- **Recommendation:** Gate the command/menu with a VS Code context key tied to the experimental setting, and do not register or open the panel when disabled. Apply an equivalent capability/feature gate in Kilo Console, or remove the experimental label and document that only the popover is controlled.

### IMPACT-003 — Medium — Enabling the prompt icon causes unsolicited marketplace networking and persistent cache writes on ordinary chat loads

- **References:** `packages/kilo-vscode/src/KiloProvider.ts:412`, `packages/kilo-vscode/src/KiloProvider.ts:421`, `packages/kilo-vscode/src/KiloProvider.ts:435`, `packages/kilo-vscode/src/KiloProvider.ts:444`, `packages/kilo-vscode/webview-ui/src/App.tsx:392`, `packages/kilo-vscode/webview-ui/agent-manager/AgentManagerApp.tsx:3140`, `packages/kilo-vscode/webview-ui/src/context/stack-summary.tsx:43`, `packages/kilo-vscode/webview-ui/src/context/stack-summary.tsx:48`, `packages/kilo-vscode/webview-ui/src/context/stack-summary.tsx:55`, `packages/opencode/src/kilocode/stack/service.ts:240`, `packages/opencode/src/kilocode/stack/service.ts:243`, `packages/opencode/src/kilocode/marketplace/service.ts:94`, `packages/opencode/src/kilocode/marketplace/service.ts:292`, `packages/opencode/src/kilocode/marketplace/service.ts:323`, `packages/opencode/src/kilocode/marketplace/service.ts:452`
- **Evidence / reproduction:** Set `kilo-code.new.experimental.projectStack=true`, then reload VS Code and open a normal Kilo sidebar or Agent Manager chat without opening the Stack wizard. `StackSummaryProvider` immediately posts `requestStackSummary` and retries after 1.5 seconds. The extension handles that by calling `load()`, which requests both stack catalog and state; both backend operations call `marketplace.manifest()`. With no cache, the default legacy endpoint resolves to `https://api.kilo.ai/api/marketplace/skills` and `/mcps`, may additionally query release URLs, and writes a manifest under the global marketplace cache.
- **User impact:** A setting described as merely showing an icon changes normal chat startup into outbound marketplace traffic and disk persistence. Multiple provider instances (sidebar, tabs, Agent Manager) can each request summaries, increasing latency and network load; offline or filtered networks generate errors for a feature the user may not actively be using.
- **Recommendation:** Populate the lightweight summary from project-local stack config without loading the marketplace manifest. Fetch marketplace data only when the wizard/catalog is explicitly opened or a preview/apply action needs it. Document any networking and cache location, and deduplicate in-flight manifest loads across instances.

### IMPACT-004 — Medium — Project Stack substantially increases shipped artifacts for all users, including duplicated eager icon assets

- **References:** `packages/kilo-vscode/esbuild.js:215`, `packages/kilo-vscode/esbuild.js:243`, `packages/kilo-vscode/webview-ui/src/App.tsx:392`, `packages/kilo-vscode/webview-ui/agent-manager/AgentManagerApp.tsx:3140`, `packages/kilo-console/src/routes/config/sections.tsx:18`, `packages/kilo-console/src/routes/config/stack/TechnologyIcon.tsx:3`, `packages/opencode/src/kilocode/stack/catalog/data.ts:8`, `packages/opencode/package.json:123`
- **Evidence / reproduction:** Build/package the extension and console on `main` and on this branch, then compare `dist`, VSIX, console static assets, and CLI binaries. The branch always builds a standalone `dist/stack.js`; also imports stack-summary/popover code into the regular sidebar and Agent Manager bundles; imports `StackRoute` into the console route table; eagerly glob-imports every dashboard icon; duplicates the icon set in the VS Code and console trees; embeds the large catalog/marketplace implementation in the CLI; and adds the `yaml` runtime dependency. The checked-in icon directories alone occupy about 388 KiB each on disk before packaging/compression.
- **User impact:** Download, installation, update, disk, and parse costs increase for every user, including those who never enable Project Stack. Console visitors can download all technology icons because the eager glob places every asset in the build graph.
- **Recommendation:** Record before/after VSIX, console asset, and CLI binary sizes as a release gate. Lazy-load Project Stack UI/routes, avoid placing Stack providers in default chat trees, replace the eager icon glob with on-demand/static URL resolution, share or deduplicate assets, and consider a generated compact catalog rather than shipping verbose source objects.

### IMPACT-005 — Medium — Shared checkbox spacing was removed globally to accommodate the wizard

- **References:** `packages/kilo-ui/src/components/checkbox.css:3`, `packages/kilo-ui/src/components/checkbox.css:22`, `packages/kilo-vscode/webview-ui/src/components/history/CloudSessionList.tsx:154`, `packages/kilo-ui/src/components/message-part.tsx:2795`, `packages/kilo-vscode/webview-ui/src/components/chat/TaskHeader.tsx:288`
- **Evidence / reproduction:** Compare any existing checkbox-heavy screen on `main` versus this branch, without opening Project Stack—for example History's “Only this repository” filter or task/todo checkboxes. The shared checkbox rule no longer supplies `margin: 4px 0`, `padding-inline-start: 10px`, or `margin-inline-end: 10px`. Existing callers were not updated to replace that spacing.
- **User impact:** Labels can sit directly against controls, rows become shorter, and adjacent content shifts throughout the existing VS Code webview UI. This is a visible regression outside the new feature and may reduce click-target separation/readability.
- **Recommendation:** Restore the shared checkbox defaults. Add wizard-specific compact spacing under a Project Stack container or expose an explicit compact checkbox variant. Visually regress existing History, task/todo, and settings checkbox surfaces.

### IMPACT-006 — Low — Stack persistence silently claims the top-level `stack` config key and a fixed project file

- **References:** `packages/opencode/src/config/config.ts:230`, `packages/opencode/src/kilocode/config/config.ts:433`, `packages/opencode/src/kilocode/stack/store.ts:142`, `packages/opencode/src/kilocode/stack/store.ts:186`, `packages/opencode/src/kilocode/stack/store.ts:207`, `packages/opencode/src/kilocode/stack/store.ts:243`
- **Evidence / reproduction:** In an existing project, place an unrelated/custom `stack` object in `.kilo/kilo.jsonc`, or use a `.kilo` symlinked directory. Start normal Kilo after upgrading, then open config diagnostics or Project Stack. The global config schema now interprets `stack` as the strict Project Stack schema, while the Stack store always targets the real project root's `.kilo/kilo.jsonc` and rejects symlinked `.kilo` paths. Applying a plan persists desired state, managed receipts, and MCP changes into that tracked project file.
- **User impact:** Even non-users can have previously tolerated config rejected because a generic key acquired product-specific meaning. Teams may also receive unexpected repository-local state and MCP config if one collaborator experiments with the wizard. Multi-root/worktree users may see different state than expected because persistence follows the resolved project root and fixed file rather than a user-selected scope.
- **Recommendation:** Namespace the feature key (for example `project_stack`), provide migration/conflict diagnostics, document the exact file/scope and source-control implications before apply, and add compatibility tests for pre-existing unknown `stack` keys and common `.kilo` layouts.

### IMPACT-007 — Low — Human verification: a broad loading-overlay selector can cover unrelated nested content

- **References:** `packages/kilo-console/src/styles/loading.css:17`, `packages/kilo-console/src/components/LoadingScreen.tsx:7`, `packages/kilo-console/src/layouts/ConfigLayout.tsx:24`, `packages/kilo-console/src/layouts/ConfigLayout.tsx:39`
- **Evidence / reproduction:** On a slow backend, open existing non-Stack Kilo Console settings and inspect nested components that render `console-loading-content`. The selector changed from direct-child `.kilo-console .content > .console-loading-content` to descendant `.kilo-console .content .console-loading-content`, so any nested content loader is now absolutely positioned relative to the outer `.content` ancestor and can cover the whole settings area rather than its local section. The new Stack route appears to motivate the change, but it affects all console content.
- **User impact:** Loading an unrelated nested panel may obscure or block the entire settings page. Confirm visually because the exact effect depends on each intermediate element's positioning context.
- **Recommendation:** Keep the direct-child rule and add a Stack-specific positioned wrapper/selector, or ensure each nested loader has an explicit local positioning container. Add a regression test for an existing settings page with nested loading state.
