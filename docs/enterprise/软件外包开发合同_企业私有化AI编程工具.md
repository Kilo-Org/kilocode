## 第一条 定义与解释

1.1 **"软件"**：指本合同项下乙方为甲方开发的全部计算机程序、源代码、目标代码、文档、数据库、配置文件及相关技术资料。

1.2 **"交付物"**：指乙方根据本合同约定应向甲方交付的全部成果，包括但不限于：
  - 源代码及编译后的可执行文件；
  - 技术文档（含设计文档、API文档、部署手册、运维手册等）；
  - 测试报告及测试用例；
  - 数据库脚本及初始化数据；
  - 部署所需的容器镜像、配置文件及 Helm Chart；
  - 培训材料及操作视频。

1.3 **"验收"**：指甲方按照本合同约定的验收标准对乙方交付物进行检验并确认符合要求的过程。

1.4 **"知识产权"**：指专利权、商标权、著作权、商业秘密及其他依法享有的专有权利。

1.5 **"开源组件"**：指乙方在开发过程中使用的、依据开源许可证（如 Apache 2.0、MIT、BSD 等）授权使用的第三方软件组件。

1.6 **"Kilo Code"**：指 GitHub 上 Kilo-Org/kilocode 开源项目，采用 Apache License 2.0 许可协议。

## 第二条 项目概述

### 2.1 项目名称
企业级私有化 AI 编程工具（基于 Kilo Code 二次开发）

### 2.2 项目背景
本项目基于 Kilo Code（Apache License 2.0）开源代码进行二次开发，面向国企、军工、航天、金融等对数据安全性要求极高的客户，提供完全私有化部署的 AI 编程辅助工具。项目核心包括：
- Layer 5 客户端层：基于 Kilo Code VS Code 扩展定制，含品牌替换、License 校验机制；
- Layer 4 API 网关层：基于 APISIX 的企业级流量治理与安全防护；
- Layer 3 企业增值层：License 引擎、RBAC 权限管理、审计日志系统、管理后台（React + Ant Design Pro）、模型配置管理；
- Layer 2 防腐桥接层：API 抽象封装、版本适配器、配置翻译层、索引服务桥接、国产模型适配层；
- Layer 1 上游 Kilo Code：Fork 后的原生代码基线，容器化引入。

### 2.3 技术栈要求

| 层级 | 技术组件 | 要求 |
|------|---------|------|
| 客户端层 | VS Code 扩展（Solid.js）、JetBrains 插件 | Fork Kilo Code，品牌替换，License 注入 |
| API 网关 | APISIX | JWT 鉴权、限流熔断、灰度发布、审计日志、SSE 流式支持、国密 SSL |
| 企业增值层 | Go（后端微服务）、React + Ant Design Pro（前端）、PostgreSQL、Redis、ClickHouse | RBAC、License 引擎、审计日志、管理后台、模型配置 |
| 桥接层 | Go | API 封装、版本适配、配置翻译、索引桥接、国产模型适配 |
| 上游基座 | Kilo Code（OpenCode Server）、Qdrant/Milvus、vLLM/SGLang | 容器化部署，多架构支持 |

### 2.4 开发范围

本合同约定的开发范围以附件《技术方案 v2.4》为基础，具体包括以下模块：

**第一批次（Phase 1：基础验证期，6 周）**
- Layer 1 容器化部署（Docker Compose 部署 Kilo Engine + Qdrant + 本地 LLM）
- Layer 2 桥接层骨架开发（Go 语言，API 封装、版本适配器基础框架）
- Layer 4 APISIX 网关部署配置（JWT 鉴权、基础限流、SSE 路由、审计日志）
- Layer 5 VS Code 插件品牌替换（Solid.js webview 组件修改、Logo/名称/配色替换）
- License 验证原型（在线校验 API + 离线 RSA 签名验证原型）

**第二批次（Phase 2：商业化核心，6 周）**
- Layer 3 License 引擎完整实现（在线/离线双模式、到期降级、数据冻结、用量计量）
- Layer 3 RBAC 权限管理 + 三员管理（5 级角色 + 三员互斥）
- Layer 3 管理后台开发（React + Ant Design Pro，8 大模块）
- Layer 2 国产模型适配层（Qwen、DeepSeek、GLM、Minimax 统一接口）
- 灰度发布与 Fallback 机制

**第三批次（Phase 3：企业级增强，12 周）**
- 项目隔离 + 向量库隔离（按项目隔离 Collection）
- 企业代码库语义索引（代码分块 → Embedding → Qdrant 写入管道）
- AI 代码安全扫描（实时规则引擎 + Semgrep 深度 SAST）
- SSO 单点登录（SAML 2.0 / OIDC / AD/LDAP 对接）
- 国密算法全栈支持（SM2/SM3/SM4）
- 审计合规体系完善（哈希链校验、WORM 只追加、180 天归档）
- JetBrains 插件开发
- 可观测性体系（Prometheus + Grafana + Loki + Jaeger）
- 规模化压测与优化

**不包含在本合同范围内的内容：**
- GPU 服务器硬件采购及机房部署（由甲方或最终客户自行负责）
- 等保测评、密评测评的第三方机构费用
- 国产 CPU（鲲鹏/海光）及国产 OS（麒麟/统信）的深度信创适配（属 Phase 4，另行协商）
- 私有模型微调及行业知识库训练

## 第五条 交付与验收

### 5.1 交付方式

乙方通过以下方式向甲方交付：
- 源代码：通过甲方指定的 Git 仓库（如 GitLab/GitHub 私有仓库）提交；
- 文档：通过邮件或甲方指定的文档管理系统交付；
- 容器镜像：推送至甲方指定的镜像仓库；
- 培训：线上或线下方式进行，不少于 8 课时。

### 5.2 验收标准

5.2.1 **功能验收**：按照附件《技术方案 v2.4》中各 Phase 的验收标准执行，所有功能点需通过测试用例验证。

5.2.2 **性能验收**：
- 代码补全流式输出延迟 P95 < 3 秒（14B 模型，100 并发）；
- 对话/Agent 响应延迟 P95 < 5 秒（80B 模型，50 并发）；
- 系统可用性 ≥ 99.9%；
- 错误率 < 0.1%。

5.2.3 **安全验收**：
- 通过甲方指定的安全扫描工具（如 Semgrep、Trivy）扫描，高危漏洞为零；
- 中危漏洞不超过 5 个，且乙方需在验收前提供修复计划；
- 依赖许可证扫描无 GPL/LGPL 传染性风险（或已隔离处理）。

5.2.4 **文档验收**：交付文档需完整、准确、可执行，包括但不限于：
- 系统设计文档（含架构图、时序图、ER 图）；
- API 文档（OpenAPI 3.0 规范）；
- 部署手册（Docker Compose / K8s Helm）；
- 运维手册（监控告警、备份恢复、应急响应）；
- 用户操作手册。

### 5.3 验收流程

5.3.1 乙方完成某批次开发后，应提前 3 个工作日向甲方提交《验收申请单》及交付物清单。

5.3.2 甲方应在收到验收申请后 **10 个工作日**内完成验收测试并出具书面验收结论。

5.3.3 验收结论分为"通过"、"有条件通过"（需限期整改）和"不通过"三种：
- **通过**：甲方签署《验收报告》，进入下一批次或支付验收款；
- **有条件通过**：甲方列明整改事项，乙方应在 **5 个工作日**内完成整改并重新申请验收；
- **不通过**：甲方列明不通过原因，乙方应在 **10 个工作日**内完成整改并重新申请验收；连续两次不通过的，甲方有权解除合同。

### 5.4 试运行

最终验收通过后，软件进入 **3 个月**试运行期。试运行期内，乙方应提供技术支持，及时修复发现的问题。试运行期满且无重大问题的，双方签署《最终验收报告》。

8.1.3 乙方应在代码提交前进行敏感信息扫描（密钥、密码、Token 等），确保不将敏感信息硬编码在代码中。

### 8.3 安全事件响应
