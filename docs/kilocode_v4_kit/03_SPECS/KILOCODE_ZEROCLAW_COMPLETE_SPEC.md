## Architecture Overview

KiloCode must not talk directly to dangerous execution without policy.

```
User -> KiloCode (cockpit) -> Hermes (mission control) -> ZeroClaw (execution)
                                                               |
                                                               v
                                                          Workspace
                                                          (isolated)

Results: ZeroClaw -> Hermes (validates) -> KiloCode (renders)
```

KiloCode owns the UI and constructs task envelopes. Hermes owns routing, policy evaluation, memory, and ledger. ZeroClaw owns sandboxed execution. KiloCode never constructs ZeroClaw jobs directly and never calls the internal Hermes-to-ZeroClaw Bridge B API.

## 2. Risk Classification Rules

KiloCode computes `risk_level` before submitting the envelope. Hermes may escalate (never downgrade).

### Classification table

| task_type | Scope: project only | Scope: system-wide | Scope: external/network | Notes |
|---|---|---|---|---|
| file_scan | low | low | low | Read-only, no side effects |
| code_generation | medium | medium | medium | Writes files but changes are buffered for review |
| code_refactor | medium | high | high | Structural changes; system-wide refactors touch more |
| test_execution | medium | high | high | Tests may have side effects (DB, network) |
| dependency_install | medium | high | high | Modifies node_modules/vendor; system-wide affects global packages |
| system_config | high | critical | critical | Modifies dotfiles, systemd units, cron, etc. |
| database_migration | high | critical | critical | Schema changes, potential data loss |
| deployment | high | critical | critical | Pushes to production |
| shell_command | medium | high | critical | Arbitrary execution; classified by scope |
| file_delete | medium | high | critical | Destructive; severity scales with scope |
| secret_rotation | high | critical | critical | Touches credentials |

### Escalation rules

1. If `network_policy=full`, bump risk by one tier (low->medium, medium->high, etc.). Cannot exceed critical.
2. If `write_policy=direct` and risk is below medium, bump to medium.
3. If `allowed_workspace_scope` includes paths outside `project_path`, bump by one tier.
4. If `approval_chain` is empty and risk is high or critical, Hermes rejects with `ZC_APPROVAL_DENIED`.

## 3. Execution Flow Diagrams

### 3.1 Low-risk flow (auto-approve)

```
    KiloCode                    Hermes                      ZeroClaw
       |                          |                            |
       |--POST /tasks----------->|                            |
       |                          |--classify: low            |
       |                          |--auto-approve             |
       |                          |--POST /jobs-------------->|
       |                          |                            |--create workspace
       |<--SSE: queued-----------|                            |--mount scope (read-only)
       |<--SSE: executing--------|                            |--execute task
       |                          |                            |--collect artifacts
       |                          |<--result + artifacts-------|
       |                          |--validate artifacts       |
       |                          |--write ledger entry        |
       |<--SSE: completed--------|                            |
       |--GET /tasks/{id}-------->|                            |
       |<--{artifacts_url}--------|                            |
       |                          |                            |
```

Wall-clock budget: under 60 seconds for typical file scan tasks.

### 3.2 Medium-risk flow (buffered writes)

```
    KiloCode                    Hermes                      ZeroClaw
       |                          |                            |
       |--POST /tasks----------->|                            |
       |                          |--classify: medium         |
       |                          |--auto-approve (if mode    |
       |                          |  allows)                  |
       |                          |--POST /jobs (buffered)--->|
       |<--SSE: queued-----------|                            |
       |<--SSE: executing--------|                            |--create workspace
       |                          |                            |--execute task
       |                          |                            |--buffer writes to tempdir
       |                          |                            |--generate diff
       |                          |<--buffered result----------|
       |                          |--validate                 |
       |<--SSE: awaiting_approval-|                            |
       |                          |                            |
       |  [User reviews diff in KiloCode UI]                  |
       |                          |                            |
       |--POST /tasks/{id}/approve->|                         |
       |                          |--POST /jobs/{jid}/apply-->|
       |                          |                            |--apply buffered writes
       |                          |                            |--verify applied state
       |                          |<--applied--------------------|
       |                          |--write ledger             |
       |<--SSE: completed--------|                            |
       |                          |                            |
```

User sees the diff panel before any writes hit disk. Cancel discards the buffer.

### 3.3 High/critical-risk flow (multi-signer approval)

```
    KiloCode                    Hermes                      ZeroClaw
       |                          |                            |
       |--POST /tasks----------->|                            |
       |                          |--classify: high/critical  |
       |                          |--block (do NOT auto-      |
       |                          |  approve)                 |
       |<--SSE: awaiting_approval-|                            |
       |                          |                            |
       |  [KiloCode shows approval modal]                     |
       |  [Each required signer reviews and approves/denies]  |
       |                          |                            |
       |--POST /tasks/{id}/approve (signer=A)-->|             |
       |<--SSE: awaiting_approval (1/2 signed)--|             |
       |--POST /tasks/{id}/approve (signer=B)-->|             |
       |                          |--all required signers done |
       |                          |--snapshot workspace------->|
       |                          |--POST /jobs-------------->|
       |<--SSE: executing--------|                            |--create workspace
       |                          |                            |--snapshot (pre)
       |                          |                            |--execute task
       |                          |                            |--collect artifacts
       |                          |<--result-------------------|
       |                          |--validate                 |
       |                          |--write ledger             |
       |<--SSE: completed--------|                            |
       |                          |                            |
```

If any required signer denies: `ZC_APPROVAL_DENIED`, task moves to `failed` state, no execution occurs.

## 4. Hermes-to-ZeroClaw Adapter Contract

KiloCode does not call this API. Documented here for completeness; Hermes owns this boundary.

### Request: Hermes -> ZeroClaw (`POST /jobs`)

```jsonc
{
  "job_id": "job-{uuid}",           // Generated by Hermes
  "source_task_id": "task-...",     // FK to KiloCode task_id
  "command": {
    "type": "shell | script | compose",
    "entrypoint": "string",         // e.g. "/bin/bash", "python3"
    "args": ["string"],
    "cwd": "string",                // Working directory inside workspace
    "env": { "KEY": "VALUE" }       // Filtered env vars (see section 5)
  },
  "workspace": {
    "mounts": [
      { "host_path": "string", "container_path": "string", "mode": "ro | rw" }
    ],
    "scratch_dir": "string",        // Temp dir for buffered writes
    "scratch_limit_mb": 512
  },
  "network": {
    "policy": "none | limited | full",
    "allowed_hosts": ["string"],    // Only used when policy=limited
    "dns_policy": "default | none"  // none = no DNS resolution
  },
  "limits": {
    "cpu": 1.0,
    "memory_mb": 512,
    "timeout_sec": 120,
    "max_output_bytes": 10485760    // 10 MB stdout+stderr cap
  },
  "write_mode": "none | buffered | direct",
  "snapshot_required": true         // If true, ZeroClaw takes pre-execution snapshot
}
```

### Response: ZeroClaw -> Hermes

```jsonc
{
  "job_id": "job-{uuid}",
  "status": "completed | failed | timeout | killed",
  "exit_code": 0,                    // Process exit code, null if killed/timeout
  "started_at": "ISO8601",
  "finished_at": "ISO8601",
  "duration_ms": 4523,
  "stdout_url": "string",           // URL to fetch stdout (internal, not exposed to KiloCode)
  "stderr_url": "string",
  "artifacts": [                     // See section 7
    { "artifact_id": "...", "type": "file", "path": "...", "size": 1024, "checksum": "sha256:..." }
  ],
  "writes_buffered": [              // Present only when write_mode=buffered
    { "path": "string", "action": "create | modify | delete", "diff": "string", "size": 1024 }
  ],
  "snapshot_id": "string | null",   // Reference to pre-execution snapshot
  "resource_usage": {
    "peak_cpu_percent": 45.2,
    "peak_memory_mb": 312,
    "disk_used_mb": 89
  },
  "error": {                        // Present only when status != completed
    "code": "string",               // ZeroClaw error code (see section 8)
    "message": "string"
  }
}
```

### Timeout handling

| Scenario | Behavior |
|---|---|
| ZeroClaw does not respond within `timeout_sec + 10s` grace | Hermes marks job as `timeout`, emits `ZC_TIMEOUT` to KiloCode |
| ZeroClaw responds with `status=timeout` | Hermes propagates. Workspace scratch is preserved for 1 hour for debugging. |
| Hermes-to-ZeroClaw network failure | Hermes retries 2 times with 5-second backoff. After 3 total attempts, `ZC_WORKSPACE_ERROR`. |

### Retry policy

| Condition | Max retries | Backoff | Notes |
|---|---|---|---|
| Network error (TCP reset, DNS fail) | 2 | 5s, 10s | Total 3 attempts |
| HTTP 429 (rate limit) | 3 | Respect `Retry-After` header, default 10s | |
| HTTP 500-503 | 2 | 5s, 10s | |
| HTTP 400/401/403 | 0 | -- | Not retryable. Report to KiloCode immediately. |
| Timeout (no response) | 1 | 15s | Single retry, then fail. |

## 5. Workspace Isolation

### Mount rules

| Write mode | Project path mount | Scratch dir | System paths |
|---|---|---|---|
| none | Read-only | Not mounted | Not mounted |
| buffered | Read-only | Read-write (all writes redirected here) | Not mounted |
| direct | Read-write | Read-write | Not mounted |

System paths (`/etc`, `/usr`, `/var`, `$HOME` outside project) are never mounted regardless of write mode.

The `allowed_workspace_scope` array from the task envelope maps to mount entries. Paths not in the array are not mounted, period.

### Environment variable filtering

ZeroClaw strips all environment variables from the execution context. Only the following are injected:

| Variable | Value | Notes |
|---|---|---|
| `PATH` | Minimal safe PATH: `/usr/local/bin:/usr/bin:/bin` | No user bin dirs |
| `HOME` | Workspace scratch dir | Not real user home |
| `TMPDIR` | Workspace scratch `tmp/` subdir | |
| `LANG` | `en_US.UTF-8` | |
| `NODE_ENV` | `production` (or as specified in task) | |
| Custom (from task `env`) | As specified | Only if key does not start with `KILO_`, `HERMES_`, `ZC_`, `AWS_`, `AZURE_`, `GCP_` |

### Secret masking

1. Before mounting any file, scan for patterns matching common secret formats (API keys, tokens, connection strings). Matches are replaced with `[REDACTED]` in the mounted copy.
2. Pattern list: `(?i)(api[_-]?key|secret|token|password|credential|connection[_-]?string)\s*[=:]\s*\S+`
3. Stdout/stderr output is also scanned before being stored. Matches are redacted in stored logs.
4. Original files on the host are never modified by the masking process.

### Temp dir lifecycle

| Event | Action |
|---|---|
| Job created | Create `scratch/{job_id}/` with 0700 permissions |
| Job executing | All buffered writes go to `scratch/{job_id}/writes/` |
| Job completed (success) | Scratch preserved for 1 hour, then deleted |
| Job completed (failure) | Scratch preserved for 24 hours for debugging, then deleted |
| Job rolled back | Scratch preserved until rollback verified, then deleted |
| Disk pressure (>90% on scratch partition) | Oldest completed-job scratches purged first |

## 6. Network Policy Enforcement

### Policy definitions

| Policy | DNS | Egress TCP | Egress UDP | Notes |
|---|---|---|---|---|
| none | Blocked | Blocked | Blocked | Complete network isolation. Localhost loopback allowed. |
| limited | Allowed (filtered) | Allowlisted hosts only | Blocked | DNS queries for non-allowlisted domains return NXDOMAIN. |
| full | Allowed | Allowed | Allowed | Unrestricted. Only for trusted tasks. |

### Allowlist format (for `limited` policy)

```jsonc
{
  "allowed_hosts": [
    "registry.npmjs.org:443",
    "pypi.org:443",
    "github.com:443",
    "*.githubusercontent.com:443"
  ]
}
```

Wildcard `*` is permitted only as the leftmost subdomain label. Port is required. If omitted, 443 is assumed.

### DNS policy

| Setting | Behavior |
|---|---|
| `default` | Use host DNS resolver but filter responses against allowlist |
| `none` | No DNS resolution. Task must use IP addresses directly. |

### Egress rules enforcement

Enforcement is implemented via network namespace + iptables rules (Linux) or equivalent per-container networking. ZeroClaw creates a dedicated network namespace per job.

| Rule | Implementation |
|---|---|
| Block all egress (policy=none) | `iptables -A OUTPUT -j DROP` in job netns (except loopback) |
| Allowlist (policy=limited) | `iptables -A OUTPUT -d <resolved_ip> -p tcp --dport <port> -j ACCEPT` per entry, default DROP |
| Log denied connections | `iptables -A OUTPUT -j LOG --log-prefix "ZC_BLOCKED: "` before final DROP. Logged to job stderr. |

## 8. Error Codes

| Code | Source | Meaning | Retryable | User-facing message |
|---|---|---|---|---|
| `ZC_TASK_REJECTED` | Hermes | Task failed validation or policy check before reaching ZeroClaw | No | "Task rejected: {reason}." |
| `ZC_TIMEOUT` | ZeroClaw | Execution exceeded `resource_limits.timeout` | No | "Task timed out after {n} seconds." |
| `ZC_RESOURCE_EXCEEDED` | ZeroClaw | CPU, memory, or disk limit hit during execution | No | "Task exceeded resource limits ({resource}: {value})." |
| `ZC_POLICY_VIOLATION` | Hermes | Task attempted action outside its policy (write in read-only mode, network in none mode, path outside scope) | No | "Policy violation: {detail}." |
| `ZC_WORKSPACE_ERROR` | ZeroClaw | Failed to create workspace, mount paths, or set up isolation | Yes (1 retry) | "Workspace setup failed. Retrying..." |
| `ZC_NETWORK_BLOCKED` | ZeroClaw | Task attempted network access denied by policy | No | "Network access blocked by policy." |
| `ZC_APPROVAL_DENIED` | Hermes | A required signer denied the task, or approval timed out | No | "Task approval denied by {signer}." |
| `ZC_ROLLBACK_FAILED` | ZeroClaw | Rollback attempted but could not restore to snapshot state | No | "Rollback failed. Manual intervention required. Snapshot: {snapshot_id}." |
| `ZC_EXECUTION_FAILED` | ZeroClaw | Task process exited with non-zero code | Depends on exit code | "Task failed with exit code {code}." |
| `ZC_ARTIFACT_CORRUPT` | Hermes | Artifact checksum mismatch on retrieval | Yes (1 retry) | "Artifact verification failed. Retrying download..." |
| `ZC_APPROVAL_TIMEOUT` | Hermes | Approval chain not completed within approval window | No | "Approval window expired ({duration})." |

### Error propagation path

```
ZeroClaw error -> ZeroClaw response.error -> Hermes validates ->
  Hermes TaskStatus.error -> SSE event (state=failed) -> KiloCode UI notification
```

KiloCode displays the `User-facing message` column. The full error (code + internal message) is logged to the "KiloCode Hermes" output channel.

## 9. Rollback Protocol

Rollback applies to high and critical risk tasks. It is optional (but recommended) for medium risk.

### 9.1 Pre-execution snapshot

Before ZeroClaw begins execution on a task with `snapshot_required=true`:

1. **File-level snapshot**: For each path in `allowed_workspace_scope` that will be mounted read-write, create a tar archive: `snapshot/{job_id}/pre.tar.gz`.
2. **Metadata record**: Store `{snapshot_id, job_id, timestamp, paths[], total_size_bytes, checksum}`.
3. **Size limit**: If total snapshot would exceed 500 MB, reject the job with `ZC_RESOURCE_EXCEEDED` and inform the user. Suggest narrowing `allowed_workspace_scope`.

### 9.2 Diff tracking during execution

While the job runs with `write_mode=direct`:

1. ZeroClaw uses an overlay filesystem (overlayfs or equivalent). The lower layer is the original mount. The upper layer captures all modifications.
2. On job completion, the upper layer diff is extracted as a list of `{path, action, before_checksum, after_checksum, size_delta}` entries.
3. This diff is stored alongside the snapshot as `snapshot/{job_id}/diff.json`.

For `write_mode=buffered`, all writes already go to scratch, so the diff is the scratch contents themselves.

### 9.3 Rollback trigger conditions

Rollback is initiated when any of the following occur:

| Trigger | Automatic | Requires user action |
|---|---|---|
| Task fails with non-zero exit code and `snapshot_required=true` | Yes (auto-rollback) | -- |
| User clicks "Rollback" in the KiloCode execution timeline | -- | Yes |
| Hermes validation of artifacts detects inconsistency | Yes | -- |
| Post-execution verification step fails (checksum mismatch on critical files) | Yes | -- |
| Approval signer triggers retroactive rollback within 1-hour window | -- | Yes |

### 9.4 Rollback execution steps

```
1. Mark task state -> "rolling_back" (new SSE event to KiloCode)
2. Load snapshot metadata for job_id
3. Verify snapshot archive integrity (checksum)
   - If corrupt: ZC_ROLLBACK_FAILED, alert user for manual recovery
4. For each path in diff.json:
   a. If action=create: delete the created file
   b. If action=modify: restore from snapshot archive
   c. If action=delete: restore from snapshot archive
5. Verify restored state:
   - Checksum each restored file against snapshot metadata
   - If any mismatch: ZC_ROLLBACK_FAILED with list of mismatched paths
6. Mark task state -> "rolled_back" (SSE event)
7. Write ledger entry: {task_id, rolled_back_at, files_restored, trigger_reason}
8. Retain snapshot for 24 additional hours, then purge
```

### 9.5 Rollback verification

After step 5, ZeroClaw performs a second pass:

| Check | Pass condition | Fail action |
|---|---|---|
| File count matches pre-snapshot | Same number of files in each scoped directory | `ZC_ROLLBACK_FAILED` with "extra files remain" |
| File checksums match | SHA-256 of each restored file matches snapshot record | `ZC_ROLLBACK_FAILED` with list of bad files |
| File permissions match | Mode bits identical to snapshot | Log warning, do not fail (permissions are best-effort) |
| No new processes spawned | No orphan processes from the rolled-back job remain | Kill orphans, log warning |

## Required UI Surfaces

These panels are rendered by KiloCode in the VS Code sidebar/editor area. Data comes from Hermes via SSE events and REST polling.

| Surface | Data source | Update mechanism |
|---|---|---|
| Execution timeline | `GET /tasks/{id}` + SSE events | Real-time via SSE, fallback poll every 5s |
| Logs panel | Artifact of type `log` | Fetched on task completion or on-demand |
| Artifact panel | Artifact list from task response | Fetched on task completion |
| Diff panel | Artifact of type `diff` or `writes_buffered` array | Shown during `awaiting_approval` state |
| Approval modal | Task status + `approval_chain` | Shown when state = `awaiting_approval` and risk >= high |
| Retry/rollback controls | Task status | Shown when state = `failed` or `completed` (within rollback window) |

## Acceptance Criteria

### AC-1: Low-risk read-only task auto-runs

| Condition | Pass | Fail |
|---|---|---|
| Submit `file_scan` task with `risk_level=low` | Task proceeds through queued -> executing -> completed without user interaction | Task blocks for approval |
| Hermes returns artifacts | Artifacts visible in artifact panel within 5 seconds of completion | Artifacts missing or delayed > 30 seconds |
| Task fails (e.g., file not found) | `ZC_EXECUTION_FAILED` shown in notification, logs available | Silent failure or no error code |

### AC-2: Medium-risk task buffers writes and shows diff

| Condition | Pass | Fail |
|---|---|---|
| Submit `code_generation` task with `write_policy=buffered` | Task executes, then state becomes `awaiting_approval` with diff visible | Writes applied without diff review |
| User clicks "Apply" on diff | Buffered writes applied, task completes | Writes not applied or applied partially |
| User clicks "Cancel" on diff | Buffered writes discarded, task state becomes `failed` with no writes on disk | Writes persist despite cancel |
| Diff panel shows file-by-file changes | Each modified file listed with additions/deletions highlighted | Single monolithic diff without file separation |

### AC-3: High-risk task blocks until approved

| Condition | Pass | Fail |
|---|---|---|
| Submit `database_migration` with `risk_level=critical` and 2-signer chain | Task stays in `awaiting_approval` until both signers approve | Task executes after first signer |
| One signer denies | `ZC_APPROVAL_DENIED`, task fails, no execution occurs | Task executes despite denial |
| Approval modal shows task description, risk level, and signer list | All fields populated and accurate | Missing or incorrect information |
| Approval timeout (configurable, default 30 minutes) | `ZC_APPROVAL_TIMEOUT`, task fails | Task hangs indefinitely |

### AC-4: Execution logs and artifacts return visibly

| Condition | Pass | Fail |
|---|---|---|
| Task produces stdout/stderr | Logs panel populated with timestamped output | Empty logs panel |
| Task produces file artifacts | Artifact panel lists files with name, size, type | Artifacts not listed |
| Artifact download | Click on artifact opens or downloads the file | Download fails silently |
| Large output (>1 MB) | Output truncated with "Show more" action | UI freezes or crashes |

### AC-5: Rollback works for failed high-risk tasks

| Condition | Pass | Fail |
|---|---|---|
| High-risk task with `snapshot_required=true` fails | Auto-rollback restores files to pre-execution state | Files left in partially modified state |
| User triggers manual rollback on completed task | Rollback executes within 5 seconds, state = `rolled_back` | Rollback not available or fails silently |
| Rollback verification fails | `ZC_ROLLBACK_FAILED` shown with details, snapshot preserved | Error swallowed, snapshot deleted |

### AC-6: Policy violations are blocked and reported

| Condition | Pass | Fail |
|---|---|---|
| Task with `network_policy=none` attempts HTTP request | Request blocked, `ZC_NETWORK_BLOCKED` in logs | Request succeeds |
| Task with `write_policy=none` attempts file write | Write blocked, `ZC_POLICY_VIOLATION` in logs | Write succeeds |
| Task accesses path outside `allowed_workspace_scope` | Access denied, `ZC_POLICY_VIOLATION` | Access succeeds |
| Secret pattern detected in task output | Pattern redacted with `[REDACTED]` in stored logs | Secret visible in logs |
